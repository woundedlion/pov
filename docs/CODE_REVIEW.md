# Holosphere вАФ Code Quality Review

*Multi-agent audit, 2026-06-09. Fifteen independent reviewers each read their component end-to-end and cross-referenced the README for intended contracts. Out of scope by request: `effects_legacy.h`, `holosphere.ino`, `rotate.h`, and the third-party `FastNoiseLite.h`. Grades are letter grades (A+ вА¶ F) and reflect quality relative to a high professional bar, not merely "does it work."*

---

## 1. Executive Summary

Holosphere is a single C++20 rendering engine that compiles to two Teensy 4.x firmware targets (a 96√Ч20 POV sphere and a four-microcontroller 288√Ч144 "Phantasm") and to a WebAssembly module that drives a Three.js browser simulator. The codebase is **mature, philosophically consistent, and exceptionally well documented**, with a fail-fast safety doctrine (`HS_CHECK`) that is not only stated but *verified by a cross-process death harness*, a 16-bit-linear / OKLCH color pipeline that reaches unbroken from the canvas to the SPI wire, and a compile-time variadic filter pipeline that auto-converts between world/screen/pixel coordinate domains.

The dominant weaknesses are uniform and unsurprising for a fast-moving solo project: **code duplication** (parallel Plot primitives, the two Trails filters, the M√ґbius family, the stereographic shader effects, the two POV-driver ISR tails), a handful of **latent edge-case guards** that the project's own fail-fast philosophy would normally trap (KDTree vertex-index truncation, `upsample` silently clamping motion-blur history, a few divide-by-(NвИТ1)/divide-by-zero paths), and **coverage gaps** in otherwise strong testing (the segmented-POV index math and the `Timeline` are untested). One effect (`SphericalHarmonics`) reimplements an engine primitive it claims to use and discards the SDF contract.

Net: the engine is **AвИТ/A work** whose remaining debt is ordinary technical debt, not architectural rot.

## 3. Per-Dimension Assessment

### Architecture & Design / Elegance вАФ **A**
The strongest dimension, and consistently so across every component. The compile-time `Pipeline<W,H,Filters...>` with recursive same-node re-dispatch on coordinate mismatch is the cleanest possible expression of "automatic world/screen/pixel conversion" and substantiates the README's "most powerful component" claim. The CRTP `ReactionDiffusionBase`, the `Transformer<вА¶,TransformFunc,вА¶>` function-pointer collapse, `WarpedVolume<Shape,Warp>` policy design with optional `correct_normal` via `requires`, the `MeshT`-generic Conway operators with shared `vertex_orbit`/`emit_vertex_orbit_faces` scaffolds, and the `SegmentController`/worker split in daydream are all genuinely elegant. Explicit `Arena&` parameter threading delivers on its no-hidden-state promise.

### Correctness & Robustness вАФ **AвИТ**
Overwhelmingly sound, with several reviewers calling out correctness reasoning that is *better documented than most production code* (the `lerp16` ARM-`smlad` sign-bit bug fix, the arena no-wrap subtractive bounds proof, the `Face` LUT Lipschitz sign-purity argument, the Canvas TOCTOU analysis). Held below A by a cluster of latent edge cases: `Polygon::distance` is a planar approximation labeled as a polygon SDF; `ComputeUVs` is used as a runtime ternary in three shapes, partially defeating the documented optimization; `Ring::calcPoint` disagrees geometrically with `Ring::sample`; `Dynamo` has a latent divide-by-zero saved only by a loop guard.

### Interface Expressiveness & API Design вАФ **AвИТ**
`FunctionRef`/`PipelineRef` zero-allocation type erasure, `normalize`/`normalized_or` encoding the fail-fast-vs-legitimate-degeneracy distinction at the type level, bool-returning failure at the JS boundary, the fluent `SolidBuilder` DSL, and the `// @ts-check` + `worker_protocol.js` typedefs that give plain JS real boundary checking are all expressive. Minor warts: two-phase `Trails::init_storage`, `Style::lerp` hand-enumerating fields.

### Readability & Naming вАФ **A**
Uniformly clear, with comments that explain *why* rather than *what*. The decomposed-into-`always_inline`-phases constructors (`Face`), the `_`-prefixed render-step methods in `driver.js`, and consistent `hs::` namespacing stand out. Deductions are localized (`Ring`'s inverted `cos_max`/`cos_min` naming, the `truncate` "50d" token that is actually a fraction).

### Documentation вАФ **A**
Among the best-documented codebases at this scale the reviewers report seeing. Load-bearing invariants are documented at the point of use: the scratch-arena contract block in `conway.h`, the `Persist<T>` member-ordering note, the relaxed-memory-order rationale in `dma_led.h`, the detached-view contract in `wasm.cpp`. The README itself is a 1,800-line reference. One overstatement: README ¬І10/¬І11 imply a JS test story the daydream repo doesn't have, and ¬І3 omits `hd107s_frame.h`.

### Performance Engineering вАФ **A**
Targeted and real for a 600 MHz MCU: split `TrigLUT`, documented `fast_*` approximations with error bounds, three-tier rasterizer culling, `BakedPalette` O(1) LUTs, `FASTRUN inline` ISR color pipeline with integer-only math, zero-allocation steady-state render loops on both the C++ and JS sides, transferable-buffer worker parallelism. The few misses are noted, not systemic (per-frame virtual `get_pixel` in the WASM readback; SSAA sub-sample loop using exact `sinf`/`cosf`).

### Error Handling вАФ **A**
The fail-fast doctrine is implemented exactly as documented and applied with discipline: `HS_CHECK` (NDEBUG-proof, `__builtin_trap`, flushed breadcrumb, no stdio) on cold seams only; stripped `assert` backed by a cold trap in per-pixel hot paths; soft bounded handling for genuinely transient conditions (DMA overrun вЖТ drop + counter, not a trap). Memory-allocation seams have complete trap coverage. The watchdog-trap on the headless `buffer_free`/DMA spins is the right call.

### Testing вАФ **AвИТ**
The death harness (re-exec child, asserts the *exact* `SIGILL`/`STATUS_ILLEGAL_INSTRUCTION`, fails CI if unrunnable) and the roster-generated construct+render+cross-run-determinism+dead-slider effect harness are unusual and excellent, and the numeric tests are regression-anchored (they read as scars from real bugs). Gaps keep it from an A: the segmented-POV branchless index math (the one place index arithmetic reaches the physical LEDs) is untested though host-testable; `Timeline` compaction/sequencing is only covered transitively; the daydream JS has no automated suite despite injectable seams.

### Concurrency / Real-Time Safety вАФ **AвИТ**
The hardware concurrency model is unusually well-reasoned: relaxed atomics justified by single-core same-observer semantics (with an explicit warning against a costly acquire/release "upgrade"), the overrun-drop protocol that avoids an ISR deadlock, equal-priority GPIO-interrupt non-preemption as the load-bearing invariant for the lock-free `x_` RMW, SysTick demotion, and frame-sync parity tracking. It rests on serialization *arguments* rather than enforced barriers вАФ correct here, but the kind of thing a commercial firmware team would want mechanically enforced вАФ and the master's self-pulse window is tolerated rather than eliminated.

### Code Duplication / Maintainability вАФ **B+**
The most consistent drag across the audit. Notable instances: parallel `Plot::` `sample()` bodies and the two `Plot::Mesh::draw` overloads; the two `Trails` filters and `World::Mobius` vs `mobius_transform`; the Liquid2D/Flyby stereographic shader skeleton (~70%); the `IslamicStars`/`HankinSolids` draw scaffold; the `pov_single`/`pov_segmented` ISR tails; `getArenaMetrics` and the four `SegmentController` broadcast loops. The codebase clearly knows the pattern (shared `pole_attenuation`/edge helpers, `correct()` shared between `load`/`packPixel`, `fragment_edge_dist`) вАФ it just hasn't been applied everywhere.

### Build / CI / Hygiene вАФ **AвИТ / B+**
CMake-preset build, opt-out pre-commit hook running the suite, presubmit CI compiling native + WASM, and a gated daydream deploy form a strong provenance chain. Hygiene nits accumulate: dead constants (`MAX_FACES`, `g_plot_cycles`), stale/contradictory comments (`ChaoticStrings`), a dead importmap entry that would 404 (`solids.html`), and the device-build `NDEBUG` (used only to strip newlib stdio, with `HS_CHECK` surviving) being subtle enough to warrant its prominent documentation.

## 4. Prioritized Fix List

### High
1. вЬЕ **`SphericalHarmonics.h` reinvents `SDF::HarmonicBlob` and discards the SDF contract** (`:11-118`, `:116`, `:158-159`). The effect ships a second nested `HarmonicBlob` + full `SHMath` namespace and returns `dist = -1.0f` unconditionally (surface faked via `frag.v1`), contradicting README ¬І9's claim that it renders "as SDF `HarmonicBlob` shapes." Feed `SHMath::sphericalHarmonic` into the engine `SDF::HarmonicBlob` via `HarmonicWaveFn` and compute a real signed distance, or rename it to a field visualizer.
   - **Fixed:** Resolved as a field visualizer, keeping the visual identical. On the unit-sphere display the rasterizer only samples r=1, so the engine `SDF::HarmonicBlob`'s lobe-radius distance would merely modulate coverage at the harmonic's nodes вАФ it never renders 3D lobes вАФ and reusing it would have cost the `acos` shortcut and the two-harmonic blend. The engine `SDF::HarmonicBlob`/`Scan::HarmonicBlob` and the `HarmonicWaveFn` typedef were dead (unused anywhere) and were removed. The effect's nested shape was renamed `HarmonicField`, its dead `amplitude` member dropped, the intentional constant distance documented, and the `(l,m)` index decode deduped. README ¬І9 now describes a colored scalar-field visualizer. Full test suite green.
2. вЬЕ **The segmented-POV index math is untested** (`hardware/pov_segmented.h`, whole file). The branchless `y_base_`/`y_step_`/`arm_b_` derivation, top-arm reversal, and `x+W/2` arm-B offset are pure host-testable arithmetic whose off-by-one silently mis-paints the sphere. Add `test_pov_segmented.h` asserting each segment's `(i вЖТ x_col, y)` mapping tiles the canvas exactly once.
   - **Fixed:** Extracted the pure index arithmetic into a dependency-free `hardware/pov_segment_map.h` (mirroring how `hd107s_frame.h` was split from `dma_led.h`), and `configure_segment()` + the column ISR now derive their mapping from those `constexpr` helpers вАФ so the new `tests/test_pov_segmented.h` exercises the real code. It asserts the per-segment arm/top-bottom derivation, the reversed bottom strip's endpoints, the arm-B `x+W/2` offset (incl. wrap), and that each segment's `(i вЖТ x_col, y)` writes tile the canvas exactly once across representative rotation columns (N=4/S=288, N=2/S=96, and an exhaustive small S=8/N=4 sweep). No behavior or performance change; suite green.

### Medium
3. **`upsample()` silently clamps motion-blur history to `CAPACITY`** (`geometry.h:535-536`, called `animation.h:968,1083`), so a fast rotation/large path jump under-samples and strobes instead of trapping вАФ the one place animation departs from the fail-fast doctrine. Add `HS_CHECK(count <= CAPACITY)` (or raise `CAP` for the affected effects).
4. вЬЕ **KDTree vertex index truncates at >65535 points** (`spatial.h:244,256`). `original_index` is `uint16_t` and assigned an untruncated `median_idx`; the node-link range is trapped but the vertex-index range is not. Add `HS_CHECK(median_idx <= UINT16_MAX)` or document the вЙ§65535-vertex precondition. *Fixed: added `HS_CHECK(median_idx <= UINT16_MAX)` matching the node-link guard.*
5. вЬЕ **`Screen::Trails` age-gating diverges from `World::Trails`** (`filter.h:639-651`): a point with `0 < age < lifetime` is seeded but not forwarded, so already-aged emissions silently vanish on the screen path. Document the intended contract or align with the world path. *Fixed: aligned with the world path вАФ `plot()` now forwards the current point unconditionally and seeds on the `ttl>0` gate, so aged emissions paint the current frame; added a regression test.*
6. вЬЕ **`MeshMorph` only morphs the incoming mesh; `mesh_A` is rendered static** (`animation.h:1451,1507-1515`), contradicting "morphs one MeshState into another" and wasting a full `MeshState` clone. Either crossfade-out the static source explicitly (drop the clone) or interpolate the outgoing geometry too.
   - **Resolved as by-design (no behavior change):** The static-outgoing/morphing-incoming split is the intended crossfade, not a defect. The incoming mesh genuinely morphs вАФ its vertices SLERP from their nearest source vertex to their dest position вАФ so the README's "interpolating positions" claim holds. The outgoing `mesh_A` is not wasted: it is rendered every frame, holding its geometry and fading via opacity (`op_A = 1 - alpha`) so the two halves sum to constant brightness across the topology swap. The separate `draw_outgoing`/`draw_incoming` callbacks (HankinSolids shades them differently) are direct evidence the split is deliberate. The source is cloned rather than borrowed to keep the animation self-contained against the caller recycling source mid-morph вАФ consistent with the borrow contract enforced on the callbacks вАФ and the transient clone is reclaimed by `compact()` after the morph. "Drop the clone" would add a source-lifetime contract that contradicts that doctrine; "morph the outgoing too" would change the visual. Documented the crossfade contract on the class instead.
7. вЬЕ **`Polygon::distance` is a planar approximation labeled as a polygon SDF** (`sdf.h:1695`) and diverges from the true geodesic distance as radius grows; **`ComputeUVs` is a runtime ternary, not `if constexpr`** in `Polygon`/`SphericalPolygon`/`Flower` (`sdf.h:1696,1802,2007`), partially defeating the documented UV-skip optimization. Document the small-cap limitation (or route large radii to `SphericalPolygon`) and gate the UV work in `if constexpr`.
   - **Fixed:** The planar behavior is by design, not a mislabel вАФ `SphericalPolygon` already exists as the geodesic counterpart, and the `Scan`/`Plot`/`ShapeShifter` layers already pair `PlanarPolygon` with `SphericalPolygon`. `SDF::Polygon` was the lone holdout, so it was renamed to `SDF::PlanarPolygon`, making the planar contract self-documenting and the SDF layer symmetric with the wrappers above it. Its doc-comment now states the tangent-plane approximation and points to `SphericalPolygon` for geodesic edges. The three `ComputeUVs ? вА¶ : 0.0f` ternaries (`PlanarPolygon`/`SphericalPolygon`/`Flower`) were converted to `if constexpr` (perf-neutral). Also corrected two stale README refs to the nonexistent `Scan::Polygon`/`Plot::Polygon`. Native suite green; WASM release builds.
8. **`HankinSolids` fragile four-`Persist` compaction + draw duplication** (`:171-182`, `:84-97` vs `IslamicStars.h:79-105`). Adopt the cleaner `carousel.compact_keep_front(...)` idiom and extract a shared topology-mesh draw helper.
9. **Liquid2D/Flyby ~70% structural duplication** (stereographic project/warp/sample/attenuate + preset-lerp boilerplate). Extract a `StereoShaderBase<W,H>` or shared free helpers.
10. вЬЕ **Device-vs-simulator RNG divergence** (`platform.h:76-79,336-337`; `pov_single.h:45`). Bare `random8()`/`random16()` resolve to the Arduino/FastLED PRNG on device but to `mt19937(1337)` on host, so the shared `1337` seed does not actually produce identical sequences device-vs-sim. Route device `random*()` through `hs::random()` (and prefer `hs::rand_*` in effects), or document that determinism is per-platform.
    - **Fixed:** Documented the contract rather than unifying the RNG, because the divergence is confined to out-of-scope code at zero practical cost. Every modern effect already draws from `hs::random()`/`hs::rand_*` (= `mt19937(1337)` on **both** device and host), so the simulator is already bit-identical to the device for everything in scope; the only callers of the divergent bare `random8/16/()` are in `effects_legacy.h` (out of scope) plus the `randomSeed(1337)` in `pov_single.h` that seeds a generator only those legacy effects consume. Routing the device's bare `random*()` through `mt19937` was rejected on performance grounds: it is strictly heavier than FastLED's LCG on the M7 for **no in-scope benefit**, and RNG is spawn/setup-time only (never per-pixel), so the LCG buys nothing measurable while breaking cross-platform determinism. Added the determinism contract to `hs::random()` (device SSOT, with a back-reference from the host mock), a note on the host `random8/16` mocks that they are reproducible-but-per-platform, and a comment on `randomSeed(1337)` clarifying it seeds only the legacy LCG path. Docs-only; no behavior or performance change.

## 5. Overall Impression & Comparative Appraisal

*This appraisal is formed from the fifteen component audits and my own synthesis, weighed against comparable work I am aware of; it is written independently and does not reuse the prior review's wording.*

What distinguishes Holosphere is not any single subsystem but the **insistence on doing everything from one disciplined source**. The same C++ that fires an `IntervalTimer` ISR on a 600 MHz Cortex-M7 вАФ packing 16-bit-linear pixels into a DMA frame with an inline color-correction pipeline, gated by a lock-free three-index double buffer whose memory ordering the author can actually defend вАФ also compiles, unchanged, to a WebAssembly module that paints an instanced Three.js sphere in a browser and reproduces the four-microcontroller partitioning *in software, before the hardware is fabricated*. Most projects pick one world and live in it. This one refuses to, and pays for the refusal with template parameterization on `<W,H>` and a platform seam (`platform.h`) that is clean enough to make the dual life look easy.

**Technical merit.** Four things stand out as genuinely above the norm. First, the compile-time filter pipeline: a variadic template that classifies each stage by coordinate domain and *recurses into itself* to lift or project a mismatched coordinate, so filter order is the only thing the author manages and the conversions are free and order-independent вАФ I would be happy to see this in a commercial renderer. Second, the color pipeline: 16-bit linear light with OKLCH perceptual, shortest-arc-hue palette interpolation, reaching unbroken from the canvas to the SPI wire with no 8-bit intermediate вАФ this is better color practice than most shipping LED products and a fair number of game engines, and the `lerp16` commentary shows the author has personally debugged the ARM SIMD path. Third, the arena allocator, whose overflow arithmetic is provably wrap-safe (a subtractive-bounds proof sits in the comments), whose lifetimes are explicit function parameters rather than hidden globals, and whose use-after-free detection uses a dual generation stamp. Fourth вАФ and rarest вАФ a fail-fast doctrine that is *documented, uniformly applied to cold paths only, and verified by a cross-process death harness that asserts the traps fire with the exact illegal-instruction status*. Stating a safety philosophy is common; mechanically proving your safety net works is not.

Set against the state of the art, the individual algorithms вАФ Gray-Scott/BZ reaction-diffusion on a Fibonacci-lattice KNN graph, the Hopf fibration, Lipschitz-bounded twisted-torus sphere tracing, authentic Hankin Islamic tilings via Conway-operator recipes, exact KD-tree spherical Voronoi вАФ are textbook-correct realizations of published techniques rather than new research. The contribution is the *assembly*: that breadth, on a sphere, on a microcontroller, behind one shading interface, with a test suite and a deterministic simulator. On engineering rigor it compares favorably to creative-coding frameworks (openFrameworks, Processing, TouchDesigner) while operating under real-time and 335 KB-arena constraints none of them face, and it is comfortably above both maker/Adafruit-tier POV builds and most academic "LED globe" demos, which rarely ship a reusable engine, a death-tested safety net, or a byte-deterministic software twin.

Where it sits below a hardened *commercial* product is breadth of validation, not depth of design. The concurrency correctness rests on serialization arguments rather than enforced barriers; the duplication would slow a team's iteration; the segmented-POV index math вАФ the one path whose arithmetic reaches the physical LEDs вАФ is untested though trivially host-testable; and a handful of edge guards (`upsample` clamp, KDTree truncation) degrade silently in exactly the way the project's own philosophy says they shouldn't. These are the items a firmware team or graphics-middleware vendor would close, and the codebase already contains the machinery to close them.

**Artistic merit.** The effects are tasteful and mathematically grounded rather than gimmicky вАФ rendering Islamic geometry via authentic Hankin construction, perceptually-uniform palette interpolation, motion blur that falls out of orientation history "for free," physically-motivated feedback styles. Judging from the documented output and the algorithms, this is gallery/installation territory, not novelty-lighting. The simulator deserves separate credit as an artistic-engineering tool: a shareable-URL, recordable, byte-deterministic web twin that also exercises the four-Teensy layout in software is something professional studios often lack.

**Bottom line.** This is **AвИТ/A** work: a cohesive, philosophically consistent, beautifully documented engine whose remaining gaps are the ordinary debt of a fast-moving solo project вАФ duplication, a few argued-rather-than-enforced invariants, and a small testing/hygiene backlog вАФ rather than any structural weakness. Clear the High and Medium items above and it would stand as a reference-quality example of how to build embedded generative-art software, and among its peer set it already ranks at or near the top on engineering rigor.
