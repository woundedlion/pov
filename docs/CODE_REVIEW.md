# Holosphere + Daydream — Code Quality Review

**Date:** 2026-06-05
**Reviewer:** Expert C++/TypeScript code audit
**Method:** 18 parallel component audits (17 subsystem readers + 1 cross-cutting pass), each reading source directly and citing `file:line`. Findings were cross-checked; the headline correctness bug was independently surfaced by two agents and then hand-verified by the lead reviewer.
**Scope:** ~33k lines of hand-written C++ engine + 27 effects + 3 hardware drivers + WASM bridge + a 22-module native test suite, across `Holosphere`; plus the ~2.4k-line vanilla-JS `daydream` web simulator. Generated tables (`reaction_graph.cpp`, `color_luts.h`) and third-party code (`FastNoiseLite.h`) were judged on integration/correctness of generation, not line-by-line.
**Out of scope (per review rules):** `effects_legacy.h` and `targets/Holosphere/Holosphere.ino`.

---

## Overall grade: **A− / B+**

This is an exceptional solo engineering project that reads, in most places, like production firmware from a top embedded studio. The architecture is principled and — rarer — *consistently applied* across a large surface: every render class is templated on `<W,H>` for zero-overhead compile-time specialization; all color is 16-bit linear light with OKLCH-correct interpolation; memory is a single partitioned arena with explicit `(Arena&, Arena&)` plumbing and no hidden state; the rendering core is a variadic `Pipeline<W,H,Filters...>` that performs automatic 2D↔3D coordinate-domain coercion at compile time; and a real fail-fast philosophy (`HS_CHECK` traps that survive `NDEBUG`) is wired into the genuine invariant sites. The performance engineering is expert-level throughout — split trig LUTs, anisotropic per-face distance LUTs with correctness-preserving validity gates, fast-reject culling, non-blocking DMA with a single cache flush, zero-copy WASM readback. The documentation (a 1,849-line README) is genuinely outstanding.

It is held back from an unqualified A by three things:

1. **One concrete, hardware-only correctness bug** *(now fixed)* — a double-applied `H_OFFSET` at the `pixel_to_vector` / SDF seam that made the float projection path clip more aggressively than the integer rasterizer path on Teensy, while looking perfect in WASM and the entire test suite (because `H_OFFSET = 0` there). This was the single most important finding in the audit.
2. **A systemic tension with the project's own fail-fast doctrine.** Roughly a dozen subsystems independently ship *silent bounded fallbacks* — return a default solid on a bad name, drop modifiers past a cap, substitute `factory[0]` for an unknown effect, swallow exceptions in empty `catch` blocks — exactly the "mask the invariant violation with plausible-but-wrong data" anti-pattern the codebase's own philosophy calls out. The discipline is applied to *allocation* faithfully but not to *lookup/registration/config*.
3. **An interface-expressiveness gap on the live-art surface.** Several effects register sliders that are inert (read once at init, or never read at all), and a few leak engine lifecycle mechanics (arena repartitioning, carousel slot bookkeeping) up into the effect body.

None of these is fatal; all are fixable; most are localized. The bones are excellent.

## The systemic theme: fail-fast applied to allocation, not to lookup

The project's stated philosophy (recorded in its own memory and the README) is unambiguous: **invariant violations must hard-fault, not degrade** — a truncated allocation or substituted default just relocates corruption into the rendered output. The engine gets this *exactly right for memory*: `Arena::allocate` (memory.h:66), `ArenaVector::push_back`/`append_bulk` (memory.h:277,288), `configure_arenas` over-subscription (memory.cpp:38), and the `vertex_orbit` anti-hang trap (conway.h:78–111) are all proper always-on `HS_CHECK` traps that survive `NDEBUG`.

But almost every *other* kind of invariant check ships a silent bounded fallback instead — and this pattern recurred independently across nine separate component audits:

| Location | Silent fallback | Should |
|---|---|---|
| `solids.h:747`, `:781` | bad index → dodecahedron; bad name → cube | trap |
| `effects_engine.h:76` (`create_effect`) | unknown effect name → `factory[0]` | log + null / trap |
| `effect_registry.h:52` (`get_fill_fn`) | any non-96×20 `<W,H>` → 288×144 fill | static_assert / generate |
| `filter.h:689` | dead `if (!dx||!dy) return` "OOM skip" (allocate can't return null) | remove |
| `canvas.h:250` (`registerParam`) | silently no-ops past 32 params | trap / static cap |
| `color.h:1064` (`AnimatedPalette::add`) | drops modifiers past `MAX_MODIFIERS=4` | trap |
| `animation.h:339` (`ParticleSystem::init`) | OOM → permanently inert no-op effect | trap |
| `spatial.h:341` (`SpatialHash::insert`) | drops entries past `MAX_ENTRIES` | trap |
| `sdf.h:461` (interval `StaticCircularBuffer`) | overflow overwrites oldest → silent missing geometry | trap |
| `segment_worker.js:161`, `daydream.js:564`, `gui.js:79` | empty `catch {}` swallows errors | surface |

Individually, each is defensible ("it's only trails", "tooling only"). Collectively they are the single largest correctness liability in the codebase, because they are precisely the regressions the test suite cannot catch — and the trap surface itself is never exercised by a death test (`tests/test_memory.h:8`). Fixing this cluster is the highest-leverage work available: it is mostly mechanical, and it converts a class of silent-wrong-output bugs into bench-time crashes, which is the entire point of the stated philosophy.

> **✅ Resolved (2026-06-05, P1 #6):** every row in the table above was converted to a trap (or a compile-time `static_assert`, or — for the JS diagnostics — a `console.warn`). See P1 #6 in the fix list for the per-site breakdown. The remaining gap is that the traps still aren't exercised by a death test; that is tracked separately under Testing.

## Standout engineering

Worth calling out because excellence should be named, not just defects:

- **The variadic filter pipeline** (`filter.h:131–193`). `Pipeline<W,H,Head,Tail...>` derives from `Head` (EBO, zero storage) and dispatches `plot()` via `if constexpr (Head::is_2d)`, transparently lifting 2D↔3D coordinates at the mismatch boundaries. A filter author writes in whatever domain is natural and the chain stitches mismatched domains together at compile time with no runtime cost. This is the cleanest possible expression of the engine's compile-time-resolution thesis.
- **The anisotropic distance LUT with a validity gate** (`sdf.h:1424–1454`). Bilinear interpolation is trusted only when all four corners share sign *and* the smallest magnitude exceeds the per-face cell-diagonal safe distance; otherwise it falls back to exact register-packed edge evaluation. Textbook-correct LUT speed without the usual interpolation artifacts near edges.
- **Compile-time borrow safety** (`animation.h:812–815`, `1323–1328`). Pairing a non-owning lvalue-capturing constructor with a `=delete`d rvalue overload turns `Motion(o, make_path(), …)` from a silent dangling-pointer-at-frame-N into a *compile error*. The lifetime contract lives in the type system at zero runtime cost.
- **The honest relaxed-atomic rationale** (`dma_led.h:294–317`). It explicitly separates the single-observer ISR/thread flag (no barrier needed) from DMA buffer coherence (handled by `arm_dcache_flush_delete` + DSB) and warns against a bogus acquire/release "fix". Staff-level reasoning that most embedded code gets wrong.
- **The end-to-end WASM detachment contract** (`wasm.cpp:114,227` ↔ `daydream.js:378`). The most commonly-botched detail of an embind zero-copy bridge — `ALLOW_MEMORY_GROWTH` detaching `typed_memory_view`s — is solved correctly at *both* ends, with the invariant documented identically on each side.
- **`SolidBuilder`** (`solids.h:184–270`). Each Conway op runs `MeshOps::op(mesh, *a, *b)` then `std::swap(a,b)`, hiding a two-arena ping-pong behind a fluent chain so Islamic-pattern recipes read as a declarative geometry DSL while staying single-allocation and arena-deterministic.

## Prioritized fix list

### P0 — Hardware correctness / memory safety

1. **✅ FIXED (2026-06-05) — `H_OFFSET` double-apply clipped inconsistently between the float and integer projection paths on Teensy.** `H_OFFSET` adds virtual rows so the image is *clipped* (not stretched) at the bottom of the ring, where the LEDs stop short of the south pole: `phi = y·π/(H_VIRT−1)`, `H_VIRT = H + H_OFFSET`, so the bottom physical row lands short of π. The integer LUT path and every SDF/scan shape divide by `(H_VIRT−1)` (correct clip), but `pixel_to_vector<W,H>(float,float)` (geometry.h:265) and `SDF::Line::get_horizontal_intervals` (sdf.h:2059) passed `H_VIRT` as the *template* argument to `y_to_phi<H>(float)`, which re-adds `H_OFFSET` internally (geometry.h:199) — dividing by `(H + 2·H_OFFSET − 1)`. So the **float projection path clipped more aggressively than the integer rasterizer path**, producing a seam/distortion between sub-pixel plotting and the LUT scan. **Invisible in WASM/tests** (`H_OFFSET=0`, platform.h:481) but real **on hardware** (`H_OFFSET=3`, platform.h:128). **Fix:** both sites now pass `<H>`, matching every other shape's bounds path (sdf.h:1504/1628/1746/1860, scan.h:185); the now-unused `H_VIRT` locals were removed. **Zero performance impact** (one fewer compile-time add). **Tests added** (test_geometry.h): `test_y_to_phi_offset_injection_clips_no_double_apply` injects the real hardware offset through the offset-parameterised free function (which the native `H_OFFSET=0` build otherwise can't exercise) to pin the clipping formula and reject the double-applied denominator; `test_pixel_to_vector_float_branch_matches_phi_lut` pins the float branch to `y_to_phi<H>`. Full native suite passes. *(Found independently by two agents; mechanism hand-verified; clipping intent confirmed by the project owner.)*
2. **✅ VALIDATED — NOT A BUG (2026-06-05). Arena bounds check is correct.** The original finding claimed `memory.h:52` mixes offset-space size with address-derived alignment padding such that a non-`max_align_t` base could let a boundary allocation overrun. Traced against `configure_arenas` (memory.cpp:40–43, which rebinds each arena to `global_arena_block + offset` with `capacity` = the byte length from that base): `padding` is a dimensionless byte count derived from the *true* address `buffer+offset`, and the *same* count is added to `offset` before comparing to `capacity`. The allocation occupies `[buffer+offset+padding, buffer+offset+padding+size)` and the arena is `[buffer, buffer+capacity)`, so it fits **iff** `offset+padding+size ≤ capacity` — exactly the check, for any base alignment. Worked example: base `0x1001`, align 8, cap 16, size 8 → padding 7, `15 ≤ 16` passes, end `0x1010 ≤ 0x1011`. The agent inverted the classic arena bug (the buggy form computes padding from the offset assuming an aligned base; this code does the robust thing). **No code change**; a clarifying comment was added at memory.h:50 to forestall the same misread.
3. **✅ FIXED / DOCUMENTED (2026-06-05) — `ArenaVector` destructor contract + `append_bulk` assert.** Two sub-issues:
   - **`append_bulk` asserted the wrong property** (`memory.h:283`): `memcpy` correctness needs *trivially-copyable*, not *trivially-destructible* (neither implies the other). Correcting the assert to `is_trivially_copyable_v` exposed that **`Vector` was not trivially copyable** — solely because it declared a redundant copy constructor (3dmath.h:107) and copy assignment (3dmath.h:127) that did plain memberwise float copies — so the existing `append_bulk` `memcpy` of `Vector` arrays (solids.h:36/41) was technically UB (benign on every real compiler). **Fix:** `= default`ed both `Vector` special members so `Vector` is trivially copyable, legitimizing the `memcpy` fast paths everywhere and letting the optimizer vectorize bulk copies (**perf-neutral-to-positive**, confirmed with the project owner); and corrected the `append_bulk` assert to `is_trivially_copyable_v`.
   - **`clear()`/move/teardown skip element destructors** (`memory.h:334`): validated as a **deliberate property of the arena model, not a bug.** An arena can be reset out from under a still-live `ArenaVector` (the reason `bind()` carries stale-binding logic), so a handle-driven destructor pass could run on already-reclaimed memory; and the only non-trivially-destructible type stored is `Fn`/`std::function` (via `EmitterFn`), whose captures are trivial, so nothing leaks on hardware (`inplace_function` never heap-allocates). A blanket `static_assert(trivially_destructible)` would not even compile. **Resolution:** documented the element-destructor contract on the `ArenaVector` class (store only types whose destructor need not run for correctness); no behavioral change. Full native suite passes.
4. **✅ FIXED (2026-06-05) — `SDF::Subtract` produced a wrong, unsorted set difference for multi-interval children.** Two coupled defects: (a) the per-A-interval shrink/split loop (sdf.h:676–711) walked B intervals in *emission* order and assumed increasing start, so an unsorted B yielded a wrong difference (e.g. `[0,100] − {[60,70],[20,30]}` wrongly emitted `[0,60],[70,100]`, dropping the `[20,30]` cut); (b) emitted pieces inherited A's order, but `scan_region`'s coalescer (scan.h:122–130) tracks a single running `current_end` and silently drops any interval whose start is out of order. Both only bite when a child emits multiple unsorted intervals (`AngularRepeat`, a nested CSG), which is why it was latent. **Fix:** insertion-sort both `intervalsA` and `intervalsB` by start before the difference loop (and before the B-empty pass-through), matching `Union`'s existing idiom — making the set difference correct and the output globally start-ordered. **Tests added** (test_sdf.h): a mock interval-emitting shape drives `test_subtract_unsorted_b_yields_sorted_set_difference` (asserts the correct 3-piece sorted result) and `test_subtract_unsorted_a_passthrough_is_sorted`; both would fail (size 2, unsorted) without the sort. No performance concern — N≤32 intervals per row, sorted once. Full native suite passes.
5. **✅ FIXED (2026-06-05, daydream repo) — `segment_worker.js` could mis-index for odd segment counts.** `computeSegmentRange` (segment_worker.js:21) computes `ySegsPerArm = floor(total/2)` and `armId = floor(id/ySegsPerArm)`; an odd `total` yields `armId ≥ NUM_ARMS` and an `x0` past canvas width → a silent degenerate band. Latent today (Phantasm uses N=4). **Fix:** the segmentation is symmetric across the 2 arms, so the count is *always* a positive even number — there can never be an odd number of segments (confirmed by the project owner; mirrors pov_segmented.h's per-arm split). Rather than distribute a remainder (which would invent an invalid layout), the function now **fails fast**: it throws if `total` is not a positive even integer, surfacing the config bug instead of rendering garbage — consistent with the engine's fail-fast posture. Verified in isolation with Node: N=2 and N=4 tile 288×144 with a clean non-overlapping cover and no OOB; odd counts (1,3,5) throw. *(daydream is a separate repo; committed there.)*

### P1 — Latent bugs & the fail-fast cluster

   - **`animation.h` `ParticleSystem::init` `is_bound()` guard** → removed as dead code (`bind()` routes through the trapping `allocate`).

   - **`spatial.h` `SpatialHash::insert`** → `HS_CHECK` trap past `MAX_ENTRIES` (was: silent drop).
   - **`sdf.h`/`scan.h` scanline interval buffers** → couldn't change `StaticCircularBuffer` globally (its evict-oldest is correct for trails), so added an `SDF::push_interval()` helper that traps on overflow and routed all 9 CSG/`scan_region` accumulation sites through it. The effect smoke harness (all 28 effects at 288×144) renders cleanly with the trap active, confirming no shipping effect exceeds 32 spans/row.
   - **JS empty `catch {}`** (daydream repo): `segment_worker.js` `getArenaMetrics`, `daydream.js` `getEffectSizes`, and both `gui.js` `updateDisplay` now `console.warn` instead of silently swallowing.

   Three unit tests that *locked the old anti-pattern* (`test_spatialhash_overflow_silently_drops`, the two solids fallback tests, `test_paramlist_capacity_cap`) were repurposed to verify the valid capacity boundary and document that overflow now traps — their failures during this work confirmed the fixes change real behavior. Full native suite passes. *(`effects_engine.h:76` in the original finding was a mislabel — `create_effect` lives in `wasm.cpp`.)*
7. **✅ FIXED (2026-06-05) — `relax` length-vs-squared-length unit mismatch** (`conway.h:813`): the degenerate-edge guard compared a vector *length* (`vec.length()`) against `EPS_LEN_SQ` (a *squared* tolerance, 1e-6). Truly-zero edges were skipped, but edges with length in `[1e-6, 1e-3]` slipped through and made `1.0f/dist` explode into a huge spring-force spike. **Fix:** compare the *squared* length (`dot(vec, vec)`) against `EPS_LEN_SQ`, matching the codebase idiom (conway.h:1064, hankin.h:266), and take the `sqrt` only for non-degenerate edges. Latent on normal meshes (edges are well above 1e-3) but a real robustness fix for near-degenerate input. **No perf trade-off** — `dot` is cheaper than `length()` and the `sqrt` is now skipped for degenerate edges. `relax` is exercised by the existing Islamic-solid generation tests (which `relax(100)`/`relax(217)` and assert unit-length + winding); full native suite passes.
8. **✅ VALIDATED — BY DESIGN, NOT A BUG (2026-06-05). `Driver` + `.then()` fires every frame.** `Driver` is `AnimationBase(1, true)`, so `done()` is true after its single per-frame `step()` and `Timeline::step` does `rewind()` + `post_callback()` each frame (animation.h:1708–1710). The original finding read this as an unintended ~60×/s callback. Confirmed with the project owner that this is **intended**: a `Driver` is a perpetual per-frame incrementer with a one-frame cycle and no natural completion, so a per-frame hook is the only coherent meaning for a `.then()` attached to it. No effect attaches `.then()` to a `Driver` today (so nothing relies on a one-shot interpretation either). **No code change**; added a doc comment on `Driver` documenting the per-frame `.then()` semantics so it isn't misread as a one-shot.
9. **✅ VALIDATED — WON'T FIX (2026-06-05). `FunctionRef` binding temporaries is the function_ref idiom, not a bug to delete.** The suggested `FunctionRef(Callable&&) = delete` would break the primary use of a `function_ref`-style type: passing a temporary lambda to a parameter that invokes it synchronously within the call (the temporary lives to the end of the full-expression, so it never dangles — this is exactly how C++26 `std::function_ref` behaves, and it deliberately does *not* delete rvalue binding). The codebase relies on this in ≥4 safe call sites — `deep_tween(trail, [&](const Quaternion&,float){…})` (ChaoticStrings.h:155, Comets.h:66, RingSpin.h:58) and `tween(history, [&](const Vector&,float){…})` (plot.h:1356), where `tween`/`deep_tween` take `TweenFn`/`VectorTweenFn` (= `FunctionRef`) by value (animation.h:1901/1917/1936). Deleting rvalue binding would force every inline shader/trail/transform lambda to be hoisted to a named local. The genuine dangling risk — *storing* a `FunctionRef` beyond the full-expression — is already mitigated at the **storage** layer: the wrappers that retain callables (`Motion`, `MeshMorph`) `=delete` their own rvalue-binding constructors, which is the correct place for the guard. The contract is also documented on `FunctionRef` (concepts.h:17–20). **No code change.**
10. **✅ FIXED (2026-06-05) — 8-bit quantization reintroduced in `ProceduralPalette::get`** (`color.h:809`): `srgb_to_linear(r_srgb * 255.0f)` narrowed float→`uint8_t`, quantizing the sRGB→linear step to 8 bits — exactly what the adjacent comment claimed to avoid. **History:** the correct full-precision float path (`65535 * srgb_to_linear_float(...)`) was added 2026-02-14 (`ea232c6`) and documented 2026-02-17 (`1a5c45d`), then **accidentally reverted on 2026-02-20 in `18685c7` ("More IslamicStars optimization")** — which swapped it for the fast 256-entry `uint8_t` LUT but left the now-contradictory comment behind. **Fix (precision-preserving, no `powf`):** added `srgb_to_linear_interp(float)` which *interpolates* the existing 256-entry LUT (lookup + lerp between the two bracketing entries) and routed `get()` through it. Full native suite passes.

## Comparative appraisal

**Versus hobbyist / maker LED-art projects (the immediate peer group):** Not comparable — this is several leagues above. The typical FastLED POV or addressable-LED art project is a single `.ino` of gamma-corrected 8-bit blends, `delay()`-driven timing, and copy-pasted effect loops. Holosphere has a typed compile-time rendering architecture, perceptual color science, an arena memory model with fail-fast invariants, a half-edge mesh kernel with the full Conway operator algebra, a WASM digital-twin simulator, and a real test suite. This is top-0.1% for the category.

**Versus professional / commercial offerings:** It compares favorably with the *engine* quality of commissioned generative-LED installation software (TouchDesigner/Madmapper-driven rigs, boutique studio firmware). Where a commercial product would pull ahead is in the operational envelope this project hasn't needed: CI on real hardware, death-tested invariants, a content-creation tool that isn't five hand-rolled HTML pages, and the disciplined removal of the silent fallbacks that a shipping product's QA would have flagged. The *core* (math, color, pipeline, memory, hardware drivers) is genuinely shippable; the *seams* (lookup fallbacks, effect param hygiene, test rigor) are where it reads as one expert's project rather than a team's product.

**Versus academic work:** The numerical maturity (matched projection sentinels, analytic Lipschitz bounds for safe sphere-tracing of warped SDFs, OKLab/OKLCH shortest-arc interpolation, the Conway-operator + Hankin-pattern geometry) is at or above the level of a strong graphics/geometry-processing course project or a SIGGRAPH-adjacent demo. What it lacks is the academic apparatus — no novelty claim, no evaluation against baselines, no ablation. It is *engineering* of known-good algorithms executed unusually well, not a research contribution. As an artifact accompanying a thesis on real-time spherical rendering it would be a standout.

**Versus the state of the art:** The individual techniques are well-chosen current practice rather than frontier (split LUTs, instanced rendering, embind zero-copy, EBO variadic templates are all known patterns). The *synthesis* — one unchanged C++ engine driving both microsecond-budget ISR firmware and a browser digital twin from a single source — is genuinely sophisticated systems design and is where the project is most impressive relative to anything off-the-shelf.

### Technical and artistic merit

**Technically**, this is the work of an expert who understands the whole stack — Cortex-M7 cache coherency, template metaprogramming, color appearance models, computational geometry, and the WASM/JS FFI — and, more unusually, who maintains a *consistent design philosophy* across all of it. The recurring weakness is not ignorance but a single inconsistency: the fail-fast rule that governs the memory layer is not enforced at the lookup/registration/config boundaries, and the test suite never proves the rule holds. Close that gap and the correctness grade moves from B to A−.

**Artistically**, the effect catalog is ambitious and coherent: reaction-diffusion on a Fibonacci-lattice graph, the Hopf fibration, procedurally-generated authentic Islamic star patterns via Hankin's method, volumetric raymarched metallic tori at the vertices of a dodecahedron, Möbius-warped wireframes — these are not stock patterns, and the engine primitives (the SDF/scan/plot families, the transformer pool, the feedback styles) are expressive enough that the strongest effects (`Raymarch`, `MeshFeedback`, `BZReactionDiffusion`, the Liquid2D/Flyby shaders) express real visual sophistication in ~25–80 lines apiece. The medium itself — a 1,200-RPM persistence-of-vision sphere painting full-color 16-bit-linear imagery — is novel and the engineering is in genuine service of the art.

**Bottom line:** A− / B+. A remarkable, near-professional codebase whose ceiling is one verified hardware bug and one consistently-applied philosophy away from an unqualified A.
