# Holosphere — Code Review

**POV LED Sphere Rendering Engine + WASM Simulator (daydream)**

> Reviewer: expert C++ / TypeScript audit. Methodology: README read in full, then **ten component sub-agents** dispatched for full-file reads of every in-scope source file across `core/`, `effects/`, `hardware/`, `targets/`, `tests/`, the `daydream/` simulator, and the build/CI infrastructure. Each subsystem graded independently on nine-to-ten software-quality dimensions; this document synthesizes those passes.
>
> **Scope:** `core/` (engine), `effects/` (28 modern effects + `ReactionDiffusionBase`), `hardware/` (4 drivers), `targets/wasm/` + `targets/Phantasm/`, `tests/`, the `daydream/` simulator (`*.js` + `tools/*.js`), and CMake/CI/git-hook infrastructure.
> **Out of scope (per rules):** `core/effects_legacy.h`, `targets/Holosphere/Holosphere.ino`.
> **Third-party / generated (graded on approach only, not line-by-line):** `core/FastNoiseLite.h`, `core/color_luts.h` (346 KB generated LUTs), `core/reaction_graph.cpp` (308 KB generated KNN table), the vendored `daydream/three.js/`.
> **Independence:** The final appraisal (§8) is formed fresh from this pass's evidence and does **not** inherit the prior review's overall-impression text.

## 1. Overall Grade: **A−**

Holosphere is a single, coherent rendering engine — compile-time `<W,H>` resolution, one 335 KB partitioned arena, 16-bit linear / OKLab color, fail-fast `HS_CHECK` traps, zero per-frame heap allocation — carried unbroken from the HD107S SPI wire, through a hard-real-time Teensy ISR, all the way to a bit-exact WebAssembly twin running in a Three.js browser scene. One codebase, three targets. The **core** (math, color, memory, mesh, the variadic filter pipeline, the hardware DMA reasoning) is genuinely top-of-domain work: numerically honest, exhaustively documented, and architecturally disciplined. It reads like A / A+ engineering.

What holds the composite at **A−** is a consistent gradient from center to edge. The computational core is excellent; the **effect layer** carries real defects (a silently-dead parameter, a doc/code mismatch, copy-paste residue, a few unbounded accumulators); the **periphery** has unenforced load-bearing invariants (a missing header include, an unbounded ISR spin with no watchdog, an inter-board sync gap, untrusted-input traps at the JS boundary the rest of the code carefully avoids); and the **test waist**, while rigorous where it reaches, leaves the `plot.h` rasterizer paths, non-trapping failure paths, and device-vs-host pixel fidelity unverified. None of these is fatal; collectively they keep an otherwise A-grade engine at A−.

## 4. Top Strengths

- **Compile-time filter pipeline** — [`core/filter.h:133-237`](../core/filter.h#L133-L237). Recursive `Pipeline<W,H,Filters...>` routes 2D/3D coordinate mismatches at compile time (`if constexpr` lifts via `pixel_to_vector` or projects via `vector_to_pixel`), validates flush contracts with `requires`-clause `static_assert`s that emit readable errors instead of deep template spew, then erases the chain behind a two-pointer `PipelineRef` so rasterizers stay non-templated and the M7 image stays small. The README's billing as the engine's best idea holds up.
- **Numerical honesty** — [`core/3dmath.h:239-938`](../core/3dmath.h#L239-L938). Every fast-math primitive (`fast_atan2`, `fast_acos`, `fast_sinf`, `fast_cbrt`) states measured error and cycle cost; the strict/graceful `normalized()` vs `normalized_or()` pair fails fast where a zero vector is a bug and gives an explicit fallback only at named geometric singularities; one documented `STEREO_INF` sentinel contract is shared across the whole stereographic/gnomonic/Möbius family with its asymmetric recognize-threshold written down.
- **Verified relaxed-atomic double buffer** — [`core/canvas.h:374-396`](../core/canvas.h#L374-L396). The three-`std::atomic<int>` (`cur_`/`next_`/`prev_`) scheme is *proven*, not asserted: strict single-writer index ownership + an idempotent ISR flip + an unconditional toggle genuinely guarantees write-buffer ≠ read-buffer with no critical section, and the sub-agent traced it end-to-end against the real ISR in [`hardware/pov_single.h`](../hardware/pov_single.h). The DMA path correctly distinguishes atomic visibility (which orders nothing for the DMA engine) from cache coherency (`arm_dcache_flush_delete` + DSB) — [`hardware/dma_led.h:117-127`](../hardware/dma_led.h#L117-L127).
- **Wrap-safe arena allocator with a use-after-free detector** — [`core/memory.h:73`](../core/memory.h#L73) uses the subtractive no-overflow bounds form (with a multiply-overflow guard upstream at [`:313`](../core/memory.h#L313)), and a two-stamp scheme (arena `generation_` catches resets, per-vector `rebind_generation_` catches re-grows, inherited by `ArenaSpan`) distinguishes the two use-after-free modes in debug builds.
- **Single-source-of-truth rosters** — [`core/effects.h:50-82`](../core/effects.h#L50-L82). One X-macro drives the include list, the WASM registry parity check ([`targets/wasm/wasm.cpp:116-117`](../targets/wasm/wasm.cpp#L116-L117)), the Phantasm firmware show list, and three test suites (smoke, determinism, param-marshal). Adding an effect without registering it is a compile or startup error, not silent drift.
- **Provenance-pinned deploy gate** — [`CMakeLists.txt:93-111`](../CMakeLists.txt#L93-L111) + `daydream/.github/workflows/deploy.yml`. The install step writes the engine commit SHA *and* the `.wasm` content hash atomically beside the module; the deploy gate verifies `sha256sum -c`, confirms the provenance trio share a last-touch commit, pins the engine checkout to that exact SHA, and runs the native suite as a hard `needs:` gate. This genuinely closes the "did the green check test the source this binary came from?" loop.
- **Out-of-process death harness** — [`tests/test_death.h:330-364`](../tests/test_death.h#L330-L364). Tests a `__builtin_trap`-based fail-fast layer via env-var child re-exec (cross-platform, no `fork()`), and refuses to count a non-`SIGILL`/non-`STATUS_ILLEGAL_INSTRUCTION` crash as a passing death test — so the safety net is verified, not assumed.

## 5. Detailed Findings by Subsystem

### 5.1 Core — Math / Color / Geometry (`3dmath.h`, `geometry.h`, `color.h`, `palettes.h`, …) — **A−**
The approximation work is rigorous and the HS_CHECK placement matches the documented doctrine. Genuine issues: the free `operator/(Vector,float)` / `operator/(Quaternion,float)` do **not** trap on zero while `operator/=` does, and `Quaternion::inverse()` routes through the untrapped path → silent NaN ([`core/3dmath.h:721`](../core/3dmath.h#L721), [`:841`](../core/3dmath.h#L841)); `fast_sinf` range reduction has a wrong-sign sliver when float rounding leaves the reduced angle marginally outside `[0,2π)` ([`:923-936`](../core/3dmath.h#L923-L936)); `Color4()` defaults `alpha=1.0` but `Fragment::color` is `Color4(0,0,0,0)` — two conflicting "default colors"; palette Modifiers silently return `t` on a null driver pointer, contradicting fail-fast ([`core/color.h:968-994`](../core/color.h#L968-L994)). Maintainability: five overloaded spellings of `y_to_phi` and a duplicated `pixel_to_vector` int/float pair.

### 5.2 Core — Rasterization / Filter Pipeline (`filter.h`, `sdf.h`, `scan.h`, `plot.h`, `styles.h`) — **A−**
Architecturally the strongest idea in the engine, but the dimension that most contradicts the project's own fail-fast creed at the numeric edges. `World::Trails` truncates TTL to `uint8_t` with no guard — a `lifetime > 255` (and the README advertises `Trails<W,50000>`) wraps silently ([`core/filter.h:436-438`](../core/filter.h#L436-L438)); `Screen::Trails` can seed a negative TTL and feed an out-of-range `t` to the trail color fn ([`:592`](../core/filter.h#L592)); `Scan::Volume`'s length-preserving `bounds_center == center` precondition is comment-only, not `HS_CHECK`ed ([`core/scan.h:795-798`](../core/scan.h#L795-L798)); `Plot::Ring::calcPoint` computes `sqrtf((1-radius)²)` — a confused no-op for `|1-radius|` ([`core/plot.h:628`](../core/plot.h#L628)). Design smell: `Pixel::Feedback` is a whole-canvas post-pass masquerading as an `Is2DWithHistory` filter — it ignores `PassFn2D` and must be terminal. Two parallel rasterizer families (`Scan::*` and `Plot::*`) overlap (Ring/Star/Flower/Polygon/DistortedRing) with subtly different register conventions.

### 5.3 Core — Animation / Transformers (`animation.h`, `transformers.h`) — **A−**
The hardest thing here — sub-frame motion-blur accumulation (collapse → `upsample` → compose → `tween`) — was traced end-to-end and is **correct**. Borrow contracts encoded as deleted rvalue overloads turn dangling-pointer hazards into compile errors across `Lerp`/`Motion`/`MobiusFlow`/`MeshMorph`. Defects: `Orientation::upsample(count)` divides by `count-1` with no guard — `upsample(1)` writes NaN across the whole history ([`core/geometry.h:528`](../core/geometry.h#L528)); `Driver` wrap corrects only one period, so `speed ≥ 1` or a resume-jump escapes `[0,1)` ([`core/animation.h:705-710`](../core/animation.h#L705-L710)); `ParticleSystem::spawn` silently drops on full, compounded by corpse-trail slot occupancy. The 78 KB single header doing ~18 animation types + particles + Timeline + the four-class Möbius family is the main maintainability drag; the `global_timeline_events` singleton (templated class, shared untemplated storage, runtime-trap-guarded) is the one leaky abstraction.

### 5.4 Core — Memory / Mesh / Geometry-gen (`memory.*`, `mesh.h`, `conway.h`, `hankin.h`, `spatial.h`, `solids.h`, …) — **A**
The best-engineered subsystem. No memory-safety bugs found in allocator math, alignment, RAII offset save/restore, KDTree edge cases, or Conway arena ping-pong. Remaining items are design smells: dead `mutable kd_tree`/`cache_valid` fields on `PolyMesh`/`MeshState` that `compute_kdtree` writes into per-call scratch through a `const` ref — the comment itself proves the cache can never be valid, so the fields only invite the dangling-into-reset-scratch bug they describe ([`core/conway.h:992-1003`](../core/conway.h#L992-L1003)); `MeshState`'s unified accessor falls through to the borrowed view when owned data is legitimately empty ([`core/spatial.h:372-393`](../core/spatial.h#L372-L393)); scattered C-style casts in `spatial.h`/`conway.h` against `static_cast` elsewhere.

### 5.5 Core — Engine glue / Reaction graph (`platform.h`, `canvas.h`, `effect_registry.h`, `reaction_graph.h`) — **A−**
The double-buffer correctness argument verifies. ✅ **Resolved (was a high-priority defect):** `canvas.h` used `assert` at three sites but never `#include <cassert>`, compiling only via transitive include from `effects_engine.h`; the direct include is now present so standalone inclusion compiles ([`core/canvas.h:415`](../core/canvas.h#L415)). The interrupt-disable in `queue_frame()` is dead weight under the single-writer model and adds avoidable per-frame ISR latency ([`:166-171`](../core/canvas.h#L166-L171)). `registerParam` has asymmetric default semantics — the float overload captures the current value as default, the bool overload *overwrites* the target ([`:331`](../core/canvas.h#L331) vs [`:341`](../core/canvas.h#L341)). `markAnimated`/`markReadonly` silently no-op on a misspelled name. `platform.h` is a 636-line catch-all with duplicated `hs::random`/`hs::log` per-platform blocks and a lying `SerialMock::printf`.

### 5.6 Effects — Group A (`ReactionDiffusionBase` + 13 effects) — **A−**
Exceptional comment discipline, real mathematical sophistication (authentic Hopf fibration with 4D tumble; BZ/GS reaction-diffusion on a 7680-node geodesic Fibonacci lattice with Wendland C2 interpolation; KD-tree spherical Voronoi proven exact). Behavioral bugs: `Moire::color_wipe` overwrites both layer palettes with the identical target, destroying the deliberate counter-rotating contrast after the first wipe ([`effects/Moire.h:72-78`](../effects/Moire.h#L72-L78)); `Metaballs` has no containment on ball position — noise can drift balls off the surface over long runs ([`effects/Metaballs.h:49-67`](../effects/Metaballs.h#L49-L67)). Duplication: the re-seed-on-GUI-change pattern is copy-pasted across Metaballs/Voronoi; `MobiusGrid` duplicates its entire ring-build loop between axis-rings and longitudes; two sibling Hankin effects solve arena compaction two different ways.

### 5.7 Effects — Group B (15 effects) — **B+**
The lowest composite. ✅ **Corrected on validation:** the originally-reported "`Liquid2D` 'Cycle Speed' is silently dead" defect does **not** hold — `Params` is a C++14 aggregate, so each preset's omitted 7th field picks up its default member initializer (`cycle_speed = 0.05f`), not zero; the breathe cycle, registered default, and `markAnimated` plumbing all worked. The genuine (milder) issue was that both preset rows listed only 6 of 7 fields, so presets could never *vary* `cycle_speed` and the code was a default-removal away from breaking — both rows now list all 7 fields explicitly ([`effects/Liquid2D.h:192-198`](../effects/Liquid2D.h#L192-L198)). ✅ **Resolved:** `MeshFeedback` loads `get_platonic_solids()` while the README claimed the Catalan library — README corrected to describe the Platonic family the code actually renders ([`effects/MeshFeedback.h:41`](../effects/MeshFeedback.h#L41)). Plus: `DistortedRing` reimplements `quintic_kernel` inline ([`effects/DistortedRing.h:70`](../effects/DistortedRing.h#L70)); `Liquid2D`/`Flyby` duplicate the entire stereo-shader skeleton (identical `attenuate`); unbounded `phase`/`accumulated_time` accumulators degrade `fast_sinf` precision over long runs. Artistically strong regardless (raymarched dodecahedral tori with metallic Blinn-Phong + Fresnel; trig-free glitch lens; attractor event-horizon masking).

### 5.8 Hardware Drivers + `platform.h` (`dma_led.h`, `hd107s_frame.h`, `pov_single.h`, `pov_segmented.h`) — **A−**
The hardest parts (DMA cache coherency, ISR double-buffer race, overrun-drop deadlock avoidance) are correct *and defended in comments*. **P1 inter-board sync gap:** the frame-sync pulse fires only at `x==0`, but arm-B's image boundary and the second `advance_display` are at `x==w/2`, so per-side (2-frames-per-rev) drift between arms is uncorrected for up to a full revolution — narrower than the README's stated guarantee ([`hardware/pov_segmented.h:364-373`](../hardware/pov_segmented.h#L364-L373)); the symmetric downstream `frame_sync_isr` re-zero can itself race the free-running column clock ([`:383-395`](../hardware/pov_segmented.h#L383-L395)). **P2:** the `waitComplete()` / `buffer_free()` spins have no watchdog — a hung DMA is a silent headless freeze, which a fail-fast codebase should trap. Telemetry: `overrunCount_` is accumulated but never surfaced, and is a non-atomic `volatile` RMW where `std::atomic<uint32_t>` belongs.

### 5.9 WASM Bridge / Targets (`wasm.cpp`, `param_marshal.h`, `Phantasm.ino`) — **A−**
The zero-copy memory-view detach contract is enforced *structurally* (buffers sized once, never resized), and `fromData` validates untrusted meshes exhaustively (length, per-index range, pre-allocation size budget). **P1 inconsistency:** `fromSolid(int)` and `fromSolidName(string)` pass untrusted JS values straight into trapping `HS_CHECK`s in `solids.h` — a stale index or typo'd name from the mesh editor aborts the entire module, exactly the outcome every neighboring entry point carefully rejects ([`targets/wasm/wasm.cpp:416-428`](../targets/wasm/wasm.cpp#L416-L428)). **P2:** `drawFrame` reads back the full canvas, ignoring the clip region, contradicting the README §10.7 claim (the JS compositor masks it). The hot readback loop does ~41k per-pixel virtual `get_pixel` calls/frame at 288×144 while the devirtualized `display_buffer()` fast path sits unused.

### 5.10 Test Suite — **A−**
Regression-driven, with a meaningful death harness and byte-exact cross-run determinism under an injected clock — it would catch real regressions. Coverage gaps are the weakness: `plot.h` draw/rasterize paths are explicitly skipped (only `sample()` helpers tested) ([`tests/test_plot_scan.h:9-11`](../tests/test_plot_scan.h#L9-L11)); non-trapping failure paths are thin outside the death harness; no golden-frame or device-vs-host pixel assertion exists anywhere (a deliberate, documented call given native/WASM `fast_*` divergence, but a real fidelity gap); ISR/DMA hand-off is single-threaded-modeled. A few tolerances are loose enough to pass a degraded implementation (`hue_rotate` gray-preservation at 48/255). `build_solid<>` is copy-pasted across three test files.

### 5.11 Daydream Simulator (JS) — **A−**
The genuinely hard interop problems are solved correctly and documented at the decision site: detached-`ArrayBuffer` handling (`byteLength === 0` re-fetch), the worker generation fence (stale frames dropped but promises still settled), pause-synced recording. **P1:** `createHolosphereModule().then(...)` has no `.catch` — a WASM load failure leaves the loading overlay spinning forever ([`daydream/daydream.js:373`](../../daydream/daydream.js#L373)). **P1:** the `composite()` zero-clear-aliasing assumption is correct only by call ordering, not structurally ([`daydream/segment_controller.js:242-249`](../../daydream/segment_controller.js#L242-L249)). `getParameterDefinitions`/`getParamValues` are unguarded against a malformed embind return throwing into the render loop; `renderer.outputColorSpace` is redundantly set every frame.

### 5.12 Build / CI / Provenance — **A−**
Above typical maker-project infra: dual-mode CMake selected by toolchain, correct `-ffast-math` + `-fno-finite-math-only` ordering (with an accurate UB rationale), least-privilege CI tokens, and the provenance-pinned deploy gate. Gaps: CI installs `clang` unpinned (`apt-get install -y clang`) while release uses the emsdk-bundled Clang — the death-harness/`__attribute__` semantics run on a different compiler than ships; `ubuntu-latest` and third-party actions (notably `mymindstorm/setup-emsdk@v14`, which provisions the toolchain) are tag-pinned, not SHA-pinned; CI never exercises the `install`/provenance path it depends on; `git rev-parse HEAD` provenance has no dirty-tree guard. **Doc defect:** README §3 advertises the engine repo as `woundedlion/Holosphere` but the real remote and the deploy gate use `woundedlion/pov` — the advertised clone URL 404s.

## 6. Prioritized Fix List

### P1 — Correctness / robustness (fix first)
1. ✅ **`Liquid2D` presets under-specify `Params`** *(validated — original "silently dead / zero-init" claim does not hold; the omitted field defaults to `0.05f` via its member initializer, so the cycle was never frozen)*. Both preset rows now list all 7 fields explicitly so presets can vary `cycle_speed` and the code no longer depends on the default member initializer ([`effects/Liquid2D.h:192-198`](../effects/Liquid2D.h#L192-L198)).
2. ✅ **`MeshFeedback` doc/code mismatch** — code cycles **Platonic** solids via `get_platonic_solids()`; corrected the README §10 entry to match the code (also fixed stale `Scan::Mesh`→`Plot::Mesh` and `Animation::Lerp`→`Animation::MeshMorph` claims). Code left as the behavioral ground truth ([`effects/MeshFeedback.h:41`](../effects/MeshFeedback.h#L41), [`README.md:1471`](../README.md#L1471)).
3. ✅ **`canvas.h` missing `#include <cassert>`** — added the direct include so any TU including `canvas.h` standalone compiles without relying on transitive includes; the three debug `assert`s at [`:415`](../core/canvas.h#L415)/[`:427`](../core/canvas.h#L427)/[`:438`](../core/canvas.h#L438) are unchanged (left as debug-only; converting to always-on `HS_CHECK` would add per-pixel cost and is out of scope).
4. **`fromSolid`/`fromSolidName` crash the WASM module on bad JS input** — validate the index range / name existence and return `nullptr`, mirroring `fromData`/`setClip` ([`targets/wasm/wasm.cpp:416-428`](../targets/wasm/wasm.cpp#L416-L428)).
5. ✅ **Segmented frame-sync drift** — the buffer flips twice per revolution (`x==0` and `x==w/2`) but the sync edge fired only at `x==0`, so a drifted board's mid-revolution frame stayed misaligned for up to a full rev. Segment 0 now pulses at both frame boundaries and downstream boards **snap `x` to the nearest boundary** (`0` or `w/2`) — tightening the drift bound to ≤ ½ rev and absorbing the small drift the sync exists to correct. The master self-pulse race was already handled (it never touches `x_`); README §"Synchronization" updated to match ([`hardware/pov_segmented.h:363-403`](../hardware/pov_segmented.h#L363-L403)).
6. **`daydream` WASM init has no `.catch`** — surface an error UI and clear the loading overlay on failure ([`daydream/daydream.js:373`](../../daydream/daydream.js#L373)).
7. **`Moire` collapses both layer palettes** on wipe — target two distinct palettes ([`effects/Moire.h:72-78`](../effects/Moire.h#L72-L78)).
8. **Untrapped scalar division** — make `operator/(Vector,float)` / `operator/(Quaternion,float)` trap on zero like `operator/=`, or document the asymmetry (`Quaternion::inverse()` silently NaNs on a zero quaternion) ([`core/3dmath.h:721`](../core/3dmath.h#L721)).

### P2 — Unenforced invariants & robustness

## 7. Cross-Cutting Observations

- **The fail-fast philosophy is real but not yet uniform.** `HS_CHECK` placement in the *core* is exemplary and matches the documented cold-path doctrine. The same discipline frays at the edges: silent Modifier null-returns, unguarded TTL truncation, an unbounded DMA spin, untrusted-JS traps in `fromSolid`. The fix is not more checks everywhere — it is closing the specific named gaps so the doctrine holds end-to-end.
- **Duplication is the dominant readability tax.** Two rasterizer families, twin stereo shaders, repeated re-seed/pixel-width/change-latch idioms in effects, copy-pasted test helpers, and parallel `getArenaMetrics` bodies. None is a bug; together they are the main thing standing between the periphery and the core's quality.
- **The test suite's rigor exceeds its reach.** Where it tests, it tests *hard* (death harness, determinism, regression-tagged invariants). The untested surfaces — `plot.h` rasterization, non-trapping failures, device pixel fidelity, ISR concurrency — are exactly the seams where the effect-layer and hardware defects above live. Closing those would convert several "found by human review" defects into "caught by CI."
- **Documentation is a genuine competitive advantage** and should be treated as a maintained asset: the few README/code drifts found (repo URL, MeshFeedback solids family, the omitted compiler flag, the clipped-readback claim) matter precisely *because* readers trust this README more than most.

## 8. Overall Impression & Appraisal *(independent assessment)*

**What this is.** A persistence-of-vision LED sphere driven by a from-scratch, header-only C++20 spherical rendering engine that targets three backends from one source — Teensy 4.x firmware with a hard-real-time column ISR, an Emscripten/WASM module, and (via that module) a Three.js browser simulator that is a bit-exact twin of the hardware. The engine ships its own SDF rasterizer, geodesic curve plotter, variadic compile-time filter pipeline, arena allocator, half-edge mesh kernel with the full Conway operator algebra, a 16-bit-linear / OKLab color stack, a timeline-driven declarative animation system, and ~28 generative effects — several of which implement research-grade algorithms.

**Technical merit.** This sits clearly in the **top tier of embedded / maker graphics work**, and on engine sophistication it is competitive with — in places ahead of — commercial LED-art products. The discriminator is the *combination*: most projects that achieve this visual sophistication do it on a desktop GPU with megabytes of RAM and a garbage collector; doing it inside a 335 KB partitioned arena on a 600 MHz Cortex-M7 with microsecond ISR deadlines, *and* keeping a browser twin bit-exact for development, is a rare engineering feat. The supporting discipline is professional-grade and in several respects above professional-average: the relaxed-atomic double buffer is proven rather than hoped, the DMA path correctly separates atomic visibility from cache coherency, the deploy gate is genuinely provenance-pinned, and the numerical primitives publish their own error bounds. Against the broad field of open-source LED/POV firmware (FastLED demos, bitmap scrollers, Instructables builds), it is not in the same category — it is an engine, not a sketch.

**Where it sits against academia and the state of the art.** The individual algorithms are drawn from the literature — Hankin's method for Islamic star patterns (cf. Kaplan), Conway/Goldberg polyhedron operators, Belousov-Zhabotinsky and Gray-Scott reaction-diffusion on geodesic graphs, the Hopf fibration, SDF sphere-tracing with Lipschitz bounds, OKLab perceptual color. The contribution is **integration and real-time embedded synthesis**, not new theory: each piece exists in research, but assembling all of them behind one zero-allocation, compile-time-specialized pipeline that runs on a spinning LED arm is novel as a *combination*. It would not be mistaken for a SIGGRAPH paper, but it is a credible, well-executed engineering realization of a dozen things that usually appear only in isolation in papers.

**Artistic merit.** The effect set is mathematically literate and, at its best, gallery-credible: an authentic 4D-tumbling Hopf fibration, reaction-diffusion running on a 7680-node Fibonacci lattice with Wendland interpolation, raymarched twisted tori at the vertices of a dodecahedron, procedurally-generated authentic Islamic geometry with ripple distortion and crossfade morphing. This is well above the visual vocabulary of typical LED art. A minority of effects (the simpler ring/shower/thruster pieces, Metaballs, Moire) are pleasant but conventional, and a handful carry the small correctness defects catalogued above — so the *curation* is a notch below the *ceiling*. But the ceiling is high, and the breadth (geometry, fluids, fibrations, volumetrics, particle systems, perceptual color) shows genuine range.

**The honest caveats.** This is an A− and not an A because the quality is not yet uniform from center to edge. The core would stand up to scrutiny in a strong professional graphics codebase; the effect layer and the untested seams would not, quite. The defects are individually minor and almost entirely concentrated where the automated tests don't reach — which is itself the most actionable finding: the gap between this project and an unqualified A is mostly the gap between its core's discipline and its periphery's, and closing the named P1/P2 items plus extending the test waist to the rasterizer and failure paths would largely erase it.

**Bottom line.** An exceptional, coherent, single-author-scale engineering achievement with real artistic ambition and the rare property of running the *same* sophisticated renderer on a microcontroller and in a browser. It substantially outclasses hobbyist POV/LED projects, holds its own against professional LED-art engines, and reads as a skilled practitioner's integration of academic techniques rather than a research contribution. **Composite: A−**, with a clearly visible, well-scoped path to A.
