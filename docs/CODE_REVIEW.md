# Holosphere — Code Quality Review

*Multi-agent audit, 2026-06-09. Fifteen independent reviewers each read one component end-to-end and cross-referenced the README for the intended contract; this report synthesizes their findings and forms its own overall appraisal. Out of scope by request: `effects_legacy.h`, `targets/*/*.ino`, `rotate.h`, and the third-party `FastNoiseLite.h`. Letter grades (A+ … F) reflect quality against a high professional/graphics-middleware bar — not merely "does it work."*

---

## 1. Executive Summary

Holosphere is a single C++20 rendering engine that compiles to two Teensy 4.x POV-sphere firmware targets (a 96×20 single-controller "Holosphere" and a four-microcontroller 288×144 "Phantasm") and to a WebAssembly module that drives a Three.js browser simulator (`daydream`). The same effect classes, arenas, and per-frame `Pixel16[]` buffer run unchanged across all three.

The audit's dominant finding is **uniformity of judgment**. The same authorial discipline recurs in every component: a hot-path-vs-cold-path rule for where invariant checks go, comments that explain *why* rather than *what*, a fail-fast `HS_CHECK` doctrine placed only on cold seams, and a 16-bit-linear / OKLCH color pipeline and compile-time `Pipeline<W,H,Filters…>` that several reviewers independently called the strongest things they saw. The safety story is not just stated — it is **mechanically verified**: a re-exec death harness asserts the *exact* illegal-instruction trap status, a determinism harness requires byte-identical re-renders, segmented-POV index math is proven to tile the canvas exactly once, and Conway operators are checked against the Euler characteristic.

The weaknesses are equally uniform, and unsurprising for a fast-moving solo project. Three themes repeat across nearly every component: **code duplication** (the stereo-shader effect pair, the preset-cycle idiom, the Plot close-loop/arc-length idiom, the two Trails filters, the Möbius family, the SSAA loop, the POV ISR tails); **README ↔ code drift** (a documented `easing::cubic_in_out` that does not compile, a `ScopedScratch` alias that does not exist, `MeshCarousel` documented as using `Lerp` but using `Sprite`, a phantom "gamma" step in the hardware pipeline); and **testing that stops at the easy side of the boundary** (the engine harness is excellent, but the most failure-prone glue — `SegmentController`'s generation fence/fault latch, the rasterizer's interval-cull-vs-brute-force equivalence — is untested). Two latent defects rise above cosmetic: `getMaxBounds` can abort the entire WASM module as the solid roster grows, and the curve clip cull tests only segment endpoints, so an arc bulging into a Phantasm board's Y-band can be silently dropped.

**Net: A− work.** A cohesive, philosophically consistent, exceptionally documented engine whose remaining debt is ordinary technical debt — duplication, a few argued-rather-than-enforced invariants, and a testing/hygiene backlog — not architectural rot.

## 3. Per-Dimension Assessment

### Architecture & Design / Elegance — **A**
The strongest and most consistent dimension. The recursive `Pipeline<W,H,Head,Tail…>` that classifies each filter by coordinate domain and re-dispatches into itself to lift (2D→3D via `pixel_to_vector`) or project (3D→2D via `vector_to_pixel`) a mismatched coordinate — with zero runtime branches and a `static_assert` that turns a non-terminal pipeline into a compile error — is the cleanest possible expression of the README's "automatic world/screen/pixel conversion" claim. It is matched by the CRTP `ReactionDiffusionBase`, the `Transformer<…,TransformFunc,…>` function-pointer pool, the half-edge Conway scaffolds (`vertex_orbit`/`emit_vertex_orbit_faces`/`for_each_edge`) that collapse a dozen operators into a few primitives, `WarpedVolume<Shape,Warp>` policy composition, the split-trig LUT (~145× memory reduction on the embedded target), and on the JS side the `SegmentController`/worker split with lazy-getter injection of a mutable engine reference. Explicit `Arena&` parameter threading delivers on its no-hidden-state promise. The lone B is Effects-Group B, where structural duplication (below) holds back otherwise-clean leaf classes.

### Correctness & Robustness — **A−**
Overwhelmingly sound, with correctness reasoning that is *better documented than most production code*: the `lerp16` round-to-nearest derivation and removal of a broken signed `smlad` dual-MAC, the arena no-wrap subtractive-bounds proof, the per-face LUT 1-Lipschitz argument, the Canvas double-buffer TOCTOU analysis, the seam-wrap normalization in `scan_region`. Held below A by a cluster of latent edges: `World::Orient` inflating `age` by +1 on a static orientation; the endpoint-only curve clip cull; `getMaxBounds`' cumulative arena growth; `ArenaVector` grow leaking the old block; `SphericalHarmonics::decode_lm` relying on `sqrtf` rounding for `m ∈ [−l,l]`; and a few unguarded reciprocals (✅ `CubemapLUT::texel_direction`, `wrap` for `m≤0`) that are safe only by construction. *`CubemapLUT::texel_direction` fixed:* the hand-rolled `/= len` was replaced by the strict `Vector::normalized()`, which traps via `HS_CHECK` if the always-≥1 length invariant is ever broken instead of dividing by zero; it runs only at LUT `build()` time, so the added guard costs nothing in the `lookup()` hot path.

### Interface Expressiveness & API Design — **A−**
`FunctionRef`/`PipelineRef` zero-allocation type erasure, `normalize()` (traps) vs `normalized_or()` (explicit degeneracy escape) encoding the fail-fast distinction at the type level, the deleted-rvalue borrow contracts on `Lerp`/`Motion`/`MeshMorph` that turn dangling references into compile errors, `add_get`'s pin flag, the fluent `SolidBuilder` DSL, bool-returning failure at the JS boundary, and `worker_protocol.js`'s discriminated-union typedefs are all expressive. Warts are minor and tracked below:

- `draw_fragments`' 8 positional args.
- `Complex` as a second-class value type.
- ✅ **`StaticCircularBuffer::emplace_*` that isn't a true emplace** (`core/static_circular_buffer.h:67,96`). Both methods did `buffer[idx] = T(args...)` — constructing a temporary and move-assigning it into the always-live `std::array` slot, so they required `T` to be assignable and were not true in-place construction. **FIXED:** added a private `construct_in_place(idx, args…)` that ends the slot's existing object lifetime (`~T()`) and placement-news the value directly in its storage, returning the reference yielded by the placement-new; `emplace_back`/`emplace_front` now route through it. The emplace path builds no temporary and needs only constructibility from `Args`, not assignability (identical codegen for trivial `T`, so the hot path is unchanged). A new `test_emplace_constructs_non_assignable_type` pins this with a deleted-assignment type that the old implementation could not have compiled.
- ✅ **`PolyMesh::initialize` dead/misleading surface** (`core/mesh.h:40-45`). The method bound only 3 of `PolyMesh`'s 4 `ArenaVector`s (skipping `topology`, so its name over-promised) and no production path called it — `to_polymesh`/`SolidBuilder`/`finalize_solid` are the real init paths, so it was a test-only convenience masquerading as production API. **FIXED:** removed the method; the test sites (two `build_solid` helpers plus the standalone fixtures in `test_conway.h`/`test_mesh.h`) now bind `vertices`/`face_counts`/`faces` directly, and `test_polymesh_initialize_binds_arrays` is renamed `test_polymesh_bind_arrays`.

### Readability & Naming — **A**
Uniformly clear and intention-revealing, with comment density that is high but almost always load-bearing. The decomposed-into-`always_inline`-phases `Face` constructor, the `_`-prefixed render-step methods in `driver.js`, the ASCII strip-layout diagrams in the POV drivers, and consistent `hs::` namespacing stand out. Localized deductions: `Dynamo::color()` is a 45-line three-way branch, `Liquid2D::apply_glitch_lens`' opaque polynomial constants, and the symbol `Q` reused with three meanings in `filter.h`.

### Documentation — **A**
Among the best-documented codebases at this scale the reviewers report seeing; load-bearing invariants are documented at the point of use (the scratch-arena contract in `conway.h`, the relaxed-memory-order rationale in `dma_led.h`, the detached-view contract in `wasm.cpp`, the determinism contract in `platform.h`). What keeps it from A+ is the recurring **README↔code drift**: `easing::cubic_in_out` (won't compile), the missing `ScopedScratch` alias, `MeshCarousel` "uses `Animation::Lerp`" (it schedules a `Sprite`), the phantom "gamma" step in the hardware correction pipeline, `DistanceResult.aux` documented as a face barycentric but always emitted as 0, the `rand_f` row of the platform table, and `setParameter`'s documented-but-unconsumed bool return. None are deep, but collectively they mean a reader navigating *from the contract* is repeatedly misled.

### Performance Engineering — **A**
Targeted and real for a 600 MHz MCU: split `TrigLUT`, documented `fast_*` approximations with measured error bounds, three-tier rasterizer culling, baked palette O(1) LUTs, `FASTRUN`/`always_inline` ISR color pipeline with integer-only math, branchless segmented ISR, coarse-grid warp + bilinear upsample avoiding a per-pixel divide+modulo, ripple fast-reject before `acos`, and allocation-free steady-state loops on both the C++ and JS sides (pooled CSS2D labels, transferable worker buffers, fixed-timestep accumulator). The misses are local, not systemic: BZ's 3× neighbor walk, `Ring::calcPoint`/`Ring::sample` on libm trig while `DistortedRing` uses a LUT, and the SSAA sub-sample loop using exact `sinf`/`cosf`.

### Error Handling (fail-fast `HS_CHECK`) — **A**
Implemented exactly as documented and applied with discipline across every C++ component: `HS_CHECK` (NDEBUG-proof, `__builtin_trap`, flushed breadcrumb, no stdio) on cold seams only — arena OOM, container growth, capacity/bounds at allocation/registration/config; stripped `assert` backed by a cold trap in per-pixel hot paths; soft bounded handling for genuinely transient conditions (DMA overrun → drop + counter). The WASM boundary correctly *rejects-and-logs* untrusted JS input rather than trapping. Two blemishes: `getMaxBounds` can hit the arena trap and abort the module (violating that file's own reject-don't-trap rule), and `daydream`'s main-thread layer silently softens the doctrine its sibling worker/layout code enforces, with no comment marking the boundary.

### Testing & Testability — **B+**
The harness itself is **A** and genuinely exceptional: a cross-process death harness that asserts the exact `SIGILL`/`STATUS_ILLEGAL_INSTRUCTION` (and hard-fails CI if unrunnable), a byte-identical determinism harness, roster-generated effect/param/determinism passes that *cannot* drift from the shipped set, Euler/manifold topology invariants across all Conway operators, a brute-force KDTree oracle, and segmented-POV tiling proofs. The dimension is held to B+ by where coverage *stops*: the rasterizer's interval-cull-equals-brute-force equivalence and Face-LUT-equals-exact are not directly asserted (though `test_sdf`/`test_scan`/`test_plot_scan` do exist — see note below); `SegmentController`'s generation fence/fault latch/composite (the simulator's riskiest, mostly DOM-free code) has no JS test; and genuine ISR concurrency is unverified by design (acknowledged honestly in `test_canvas.h`).

> **Reconciliation:** The Mesh and Rasterization reviewers each reported "no tests" for their components, but they reviewed only `core/` headers. The Test-suite reviewer confirms `test_conway.h` (Euler V−E+F=2, manifold edge-pairing, winding across four seeds and all 13 operators), `test_hankin.h`, `test_mesh.h`, `test_solids.h`, `test_sdf.h`, `test_scan.h`, and `test_plot_scan.h` all exist and pass. The accurate statement is narrower: these subsystems *are* tested, but the specific *property* tests the reviewers wanted (interval cull == full-row scan; LUT bilinear == exact within tolerance) are the residual gap.

### Code Duplication / Maintainability — **B+**
The most consistent drag across the audit, and the clearest signal of a fast solo project. Notable instances: `Liquid2D`/`Flyby` share ~the entire stereo-shader skeleton with no base class; the "preset-cycle with pause-gate + `Lerp(&anims_paused_)`" idiom and three hand-written `Params::lerp()` bodies are copy-pasted across four effects; the Plot manual-close + cumulative-arc-length idiom recurs across seven primitives; the two `Scan::Shader` SSAA loops, the two `Trails` filters, `World::Mobius` vs `mobius_transform`, `OrientationTrail`/`VectorTrail`, the `pov_single`/`pov_segmented` ISR tails, `add()`/`add_get()`, and the four mesh test fixtures are all near-duplicates. The codebase clearly knows the pattern (shared `pair_half_edges`, `correct()`, `collect_arena_metrics`, `apply_vertex_shader`) — it just hasn't been applied everywhere.

### Concurrency / Real-Time Safety — **A−**
The hardware and Canvas layers are a standout: relaxed atomics justified by single-core same-observer semantics (with the cache-flush handling DMA coherence separately, and an explicit warning against a costly acquire/release "upgrade"), the overrun-drop protocol that avoids an ISR deadlock, equal-priority GPIO-interrupt non-preemption as the load-bearing invariant for the lock-free `x_` RMW, SysTick demotion, and frame-sync parity tracking. The `daydream` worker pipeline (generation fence + fault latch) is equally careful. The dimension is held to A− by the math layer: the lazy `if(!initialized) init()` LUT guards are a non-atomic check-then-init whose "self-healing fallback" framing is slightly stronger than the code supports — the safety rests entirely on eager init being called at every production entry point, and that single-thread assumption is undocumented at the lazy call sites.

## 4. Prioritized Fix List

### High
1. ✅ **`getMaxBounds` accumulates every solid into the 8 MB tooling arena with no per-iteration reset** (`targets/wasm/wasm.cpp:705-727`). The loop resets the two scratch arenas but never `tooling_arena`, and each `Solids::get` finalizes its mesh into it cumulatively. As the solid roster grows, the summed size can exceed 8 MB and the arena's fail-fast trap **aborts the entire WASM module** — the exact outcome the rest of the file avoids with reject-and-log guards. It also leaves the arena dirty on return. *Fixed:* `tooling_arena.reset()` now runs at the top of each iteration (solids are measured independently) and once more before return.
2. ✅ **Curve clip cull tests only segment endpoints; arcs bulging into a clip band are dropped** (`core/plot.h:339-344`, `constants.h:57-61`). A great-circle or planar-projected segment is not latitude-monotonic — an arc between two points below a clip band can rise through it toward a pole. In the Phantasm 4-board partition each board renders a Y-band (README §7.10), so a curve crossing a board boundary can show a **gap in production hardware**, not just the simulator; `Line`/`Mesh` long chords are most exposed. *Fixed:* the cull now widens each edge's screen-row span to the arc's true latitude extremum via a new `edge_row_span()` (`core/plot.h`). For geodesic edges the extremum is closed-form — the great circle's peak latitude `sqrt(1 − n.y²)`, applied only when the forward-tangent y-component flips sign across the edge (i.e. the turning point lies inside the span); this is *exact* and actually cheaper than the old two `vector_to_pixel` calls (it skips the azimuth `atan2`). For planar (azimuthal-equidistant) edges, which are not great circles, it samples the arc with the renderer's own unprojection and widens by the arc's Lipschitz bound in row space (`phi` is 1-Lipschitz in angular distance; the projected chord over-estimates the on-sphere arc), making the span provably gap-free. The cull runs only when a clip is active (segmented mode), so full-canvas rendering is untouched; a randomized property test (`tests/test_plot_scan.h::test_edge_row_span_covers_arc_bulge`) asserts the span covers a densely-sampled arc across thousands of geodesic and planar edges. Validation measured the old endpoint-only test false-culling 40–83% of bulging edges. The planar `azimuthal_project/unproject` math was also factored into shared helpers, removing the duplicated projection in `rasterize_planar_strategy`.

### Medium
3. ✅ **`World::Orient` inflates `age` by +1 every frame even for a static orientation** (`core/filter.h:285-288` via `tween` at `animation.h:2183-2190`; `OrientSlice` has the same pattern). `tween` with `num_frames==1` runs once at `t=0`, emitting `age + 1`, contradicting the documented "only the SLERP sweep adjusts age" contract and shifting the temporal channel for any downstream age-driven palette or post-`Orient` `Trails`. No test pins the value. *Fix:* emit `t=1.0` (age-neutral) for the single-frame case, or document the +1 as a deliberate one-frame integration offset and add a pinning test. **FIXED:** the `tween(Orientation)` helper now emits `t=1.0` for the lone-snapshot case (it is the newest sub-position), so a static orientation is age-neutral; `test_world_orient_rotates_and_offsets_age` updated to pin `age` unchanged.
4. ✅ **BZReactionDiffusion runs 3 separate neighbor walks per node, violating its own base class's guidance** (`effects/BZReactionDiffusion.h:158-160` vs `ReactionDiffusionBase.h:80-82`). `step_physics` calls `graph_laplacian` once per species, re-reading the K=6 adjacency 3× per node per substep on the effect's dominant loop; `GSReactionDiffusion` already does the recommended single fused walk. *Fix:* fuse into one `for_each_neighbor` accumulating lapA/lapB/lapC, then call `advance_species`. **FIXED:** `step_physics` now accumulates lA/lB/lC in a single `for_each_neighbor` walk (mirroring GS); the per-species Σ runs over the same neighbor order, so output is bit-identical (determinism suite passes) while the lattice is read once instead of three times.
5. ✅ **`ArenaVector` grow silently leaks the old block with no diagnostic** (`core/memory.h:300-331`). Re-`bind()` to a larger capacity abandons the old block until the next reset — a supported pattern (HankinSolids morph), but on a 335 KB device with no reset between morphs it is real silent consumption, observable only indirectly via the high-water mark. *Fix:* add an `#ifndef NDEBUG` abandoned-bytes counter or an `hs::log` on the lossy grow path, consistent with the project's fail-loud habit. **FIXED:** `bind()` now emits an `#ifndef NDEBUG` `hs::log` on the genuine lossy-grow path (reached only with a still-live binding to the same arena/generation), reporting the abandoned byte count and old→new capacity; `test_arenavec_rebind_grows` adds coverage for the previously-untested grow branch.
6. ✅ **`ShapeShifter` Sprite lambdas capture references into `ArenaVector` storage; safety rests on undocumented address stability** (`effects/ShapeShifter.h:99-108`, `:71-92`). Correctness depends on `ArenaVector::clear()` preserving slot addresses and on `timeline.clear()` running before `shapes.clear()`. Both hold today; neither is asserted. *Fix:* capture a stable index into `shapes` and re-index at draw time, or document + `static_assert` the dependency and the clear ordering. **FIXED:** `rebuild()` now clears the timeline *before* `shapes` (was reversed), so the capturing animations are dropped before their target slots are destroyed; the address-stability contract (fixed-capacity bind, never re-bound; `clear()` resets size only; the capacity guard is load-bearing) is documented at the bind site, the capture site, and the teardown. Index capture was rejected because `Rotation` binds `ring.orientation` by reference and shares the same dependency.
7. ✅ **`daydream` fail-fast doctrine flips between layers with no documented boundary** (`daydream.js:186-189,322-325,349-350`; `segment_worker.js:169-174`). The pure/worker layer throws (`segment_layout`, `composite`); the main-thread UI layer downgrades `setEffect`/`setResolution`/`getEffectSizes`/`getArenaMetrics` failures to `console.error/warn`. Defensible (don't white-screen the user), but unmarked. *Fix:* a one-line doctrine note stating UI failures degrade gracefully while engine/protocol invariants trap. **FIXED:** added a doctrine banner at the top of `daydream.js` (daydream repo) stating the rule — the UI layer degrades gracefully on recoverable user/config input, the engine/protocol/pure layers trap on violated invariants — so the graceful catches no longer read as missing error handling.
8. ✅ **`SegmentController`'s highest-risk logic is untested** (`segment_controller.js:164,245-257,318-404`). The generation-fence drop, fault-latch deadlock-break, and quadrant composite are the simulator's most failure-prone code and are largely DOM-free. *Fix:* drive a mock `Worker` through `onmessage`/`onerror`; assert stale-generation frames drop, a fault zeroes `pending` and resolves the frame, and `composite()` blits to the right offset / rejects an out-of-bounds rect. **FIXED:** added `tests/segment_controller.test.js` (daydream repo) — a fake `Worker` (captured `postMessage`, hand-invoked `onmessage`/`onerror`) plus a mocked `./driver.js` so Node never loads the three.js/lil-gui chain. Eight tests cover matching- vs stale-generation frames (dropped but still settled), the fault latch (zeroes `pending`, releases `renderInFlight`, resolves the frame, records only the first fault, cleared by `destroy()`), and the compositor (correct-offset blit, out-of-bounds-rect skip, alias-break throw). The `npm test` script now enables Node's experimental module mocks.
9. ✅ **Resolve the README↔code drift cluster** (docs-only but compile-breaking in one case). `easing::cubic_in_out` does not exist and the README example won't compile (`README.md:831` vs `easing.h`); the `ScopedScratch` alias is documented but absent (`README.md:889`); `MeshCarousel` is documented as using `Animation::Lerp` but schedules a `Sprite` (`README.md:775` vs `animation.h:2123`); the hardware pipeline lists a phantom "gamma" multiply (`README.md:1135` vs `hd107s_frame.h:97-116`); `DistanceResult.aux` is documented as a face barycentric but `Face` always emits `0.0f` (`sdf.h:1619`); the `rand_f` platform-table row understates that it's deterministic on both targets; `setParameter`'s bool return is documented but never consumed. *Fix:* either add the missing alias / correct the example, or update each contract to describe the current state. **FIXED:** README updated to current state — `ease_in_out_bicubic` in the example; `ScratchScope` only (dropped the absent-alias parenthetical); `MeshCarousel` schedules an `Animation::Sprite` crossfade; the hardware pipeline reads color → temperature → brightness (no gamma); `DistanceResult.aux` described as a shape-dependent secondary parameter (0 for faces); the `rand_f` row shows the shared seeded `std::mt19937(1337)` on both targets. `setParameter`'s bool return is now consumed at the daydream GUI call sites (warn on rejection).

### Low
10. ✅ **Rasterizer interval/​LUT property tests** (`core/sdf.h:359-374`, `:1585-1615`). Add: for each shape over a grid of orientations/radii, assert the interval-cull pixel set equals a brute-force full-row distance scan; and `|LUT bilinear − exact| < cell-diagonal` for sign-pure cells. (Tests exist for the rasterizer broadly; this specific equivalence is the gap.) **FIXED:** `test_sdf.h` gains two property tests driven by independent brute-force oracles. (a) *Interval cull*: over a 7-orientation × 3-radius grid for Ring/SphericalPolygon/Star/PlanarPolygon, every pixel clearly inside the shape (`dist < −pixel_width`) is asserted present in the exact set of pixels `scan_region` visits — pinning that the two-tier cull never drops a covered pixel. (b) *Face LUT*: validating the conjectured `< cell-diagonal` magnitude bound showed it is **false** (worst case ~1.3× the cell diagonal for a pentagon, ~4× for a sharp-vertexed triangle — `lut_safe_dist` bounds distance to the zero set, not to the vertex-fan medial creases). The test instead pins the two invariants that actually govern rendering and *do* hold: the LUT path's **sign is always exact** (sign-purity guard), and it **never serves a near-boundary magnitude** (every LUT result ≥ ~cell-diagonal, so it lands outside the ±pixel_width AA ramp and only its sign matters).
11. ✅ **Inconsistent fast vs full-precision trig in shape constructors** (`core/sdf.h:1649,1651,1759,1862`): `PlanarPolygon`/`SphericalPolygon`/`Star`/`Flower` use `atan2f`/`acosf` for cull bounds while `Ring`/`DistortedRing` use `fast_*` for the identical quantities. Standardize on `fast_*` (or route through `phi_bounds_to_rows<H>`, which they also open-code). **FIXED:** standardized on **full-precision** trig, not `fast_*` — these are constructors (cold path, once per shape, never per pixel), so the polygon family's exact `acosf`/`atan2f` is the right baseline. `Ring`/`DistortedRing` now use clamped `acosf` (matching the polygon ctors) for `center_phi`, `atan2f` for `alpha_angle`, and `cosf`/`sinf` for the precomputed band/centerline constants. Those constants feed the per-pixel reject and centerline distance, so making them exact costs nothing in the hot loop while removing the ~0.1–0.2% approximation from the SDF; full test suite passes.

## 5. Overall Impression & Comparative Appraisal

*Formed independently from the fifteen component audits and my own synthesis, weighed against comparable work; written without reusing any prior review's wording or conclusions.*

The most striking property of this codebase is not any single subsystem — it is that **one person's engineering taste is legible in every file**. Fifteen reviewers working blind on different components reported the same things in the same words: a deliberate hot-path/cold-path split for where checks live, comments that argue rather than narrate, fail-fast traps confined to cold seams, and a refusal to silently degrade where degradation would corrupt rather than merely dim. That kind of consistency across ~30 K lines spanning fixed-point SIMD color math, half-edge polyhedral topology, lock-free ISR double-buffering, and a Web Worker render pipeline is rare and is itself a quality signal: the engine was clearly built by someone holding the whole thing in their head, applying a small number of load-bearing ideas relentlessly.

**Technical merit.** Three things are genuinely above the professional norm. The first is the compile-time filter pipeline — a variadic template that tags each stage by coordinate domain and recurses into *itself* to lift or project a mismatched coordinate, so the author manages only filter order and the conversions are free, order-independent, and guarded against non-termination by a `static_assert`. I would be glad to find this in a shipping renderer. The second is the color path: 16-bit linear light with perceptual OKLCH, shortest-arc hue interpolation, reaching unbroken from the canvas to the SPI wire with no 8-bit intermediate, and a `lerp16` whose author has personally debugged the ARM `smlad` sign-bit failure and left the proof in the comments — this is better color practice than most shipping LED products and a fair number of game engines. The third, and rarest, is that the project *tests its own marketing*: the fail-fast doctrine is not asserted but verified by a cross-process death harness that checks the exact illegal-instruction status; determinism is verified by byte-identical re-renders under an injected clock; the one path whose arithmetic reaches the physical LEDs (segmented-POV index math) is proven to tile the canvas exactly once; and the Conway operators are checked against the Euler characteristic. Stating a safety property is common. Mechanically proving the safety net fires is not.

Set against the state of the art, the *algorithms* — Gray-Scott/BZ reaction–diffusion on a Fibonacci-lattice KNN graph, the Hopf fibration, Lipschitz-bounded twisted-torus sphere tracing, authentic Hankin Islamic tilings via Conway-operator recipes, exact KD-tree spherical Voronoi — are faithful, well-implemented realizations of published techniques, not new research. The contribution is the *assembly*: that breadth, on a sphere, on a microcontroller, behind a single shading interface, with a verified test suite and a byte-deterministic software twin that also reproduces the four-board hardware partition before fabrication. On engineering rigor it compares favorably to creative-coding frameworks (openFrameworks, Processing, TouchDesigner) while operating under real-time and 335 KB-arena constraints none of them face, and it is well above both maker/Adafruit-tier POV builds and most academic "LED globe" demos, which rarely ship a reusable engine, a death-tested safety net, or a deterministic simulator.

Where it sits below a hardened *commercial* product is breadth of validation and uniformity of execution, not depth of design. The duplication would slow a team's iteration and is the single thing most likely to cause a future divergence bug (a fix applied to one of two near-identical copies). The README has drifted from the code in enough places that a newcomer navigating from the documentation is repeatedly tripped — including one example that does not compile. The concurrency correctness, though unusually well-reasoned, rests on serialization *arguments* rather than mechanically enforced barriers. And the testing, excellent at the engine core, stops short of the glue that is statistically most likely to break — the segmented-POV controller and the curve clip cull, the latter of which can mis-render real hardware. These are precisely the items a firmware team or graphics-middleware vendor would close in review, and the codebase already contains the machinery (the property-test harness, the pure-logic extractions, the shared scaffolds) to close them.

**Artistic merit.** The effects are mathematically grounded and tasteful rather than gimmicky — authentic Hankin construction for Islamic geometry, perceptually uniform palette interpolation, motion blur that falls out of orientation history "for free," physically motivated feedback styles. Judging from the documented output and the algorithms, this is installation/gallery territory, not novelty lighting. The simulator deserves separate credit as an artistic-engineering instrument: a shareable-URL, recordable, byte-deterministic web twin that also exercises the four-Teensy layout in software is something professional studios often lack.

**Bottom line.** This is **A− work**: a cohesive, philosophically consistent, beautifully documented engine whose remaining gaps are the ordinary debt of a fast-moving solo project — duplication, a handful of argued-rather-than-enforced invariants, documentation drift, and a focused testing backlog — rather than any structural weakness. Close the two High items (the WASM module-abort and the production-affecting clip cull), drain the duplication and README-drift backlogs, and it would stand as a reference-quality example of how to build embedded generative-art software; among its peer set it already ranks at or near the top on engineering rigor.
