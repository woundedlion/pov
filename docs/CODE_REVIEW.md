## Overall grade: **AŌłÆ / B+**

This is an unusually sophisticated and well-engineered codebase for what is effectively a solo generative-art project. The architecture is principled and consistent (compile-time `<W,H>` specialization, 16-bit linear color, partitioned arena allocation, a variadic filter pipeline with automatic coordinate-domain coercion), the performance engineering is genuinely expert-level (LUT-driven trig, fast-reject heuristics, zero-copy WASM readback, non-blocking DMA, ISR double-buffering), and the documentation is exceptional. The project also has a clearly-articulated and *largely well-executed* **fail-fast philosophy**: invariant violations hard-fault via an always-on `HS_CHECK` trap that survives `NDEBUG`, while genuinely transient conditions degrade gracefully.

It is held back from an unqualified A by three things: (1) a **pixel-sink hardening gap** where projected/fractional coordinates can in principle write out of bounds on the `NDEBUG` device before any trap fires; (2) a recurring **`assert`-vs-`HS_CHECK` inconsistency** ŌĆö several invariant checks still use `assert`, which is stripped on device, partially undermining the otherwise-excellent fail-fast posture; and (3) a small cluster of **concrete correctness bugs** ŌĆö two 16-bitŌåÆ8-bit color truncations in shipping effects, a WASM format-string defect, three effects that abort at 288├Ś144, and a couple of dangling-reference hazards in the animation layer.

## Failure philosophy (the standard these findings are measured against)

This codebase runs on hardware in a **fail-fast** posture, and the recommendations below assume it:

- **Invariant violations must hard-fault, not degrade.** Arena over-allocation, container capacity overflow, OOM, or an out-of-bounds index is a logic/sizing bug with no valid recovery ŌĆö a truncated or aliased allocation just relocates the corruption into the rendered output. These must crash at the violation site so they're caught on the bench, not ship silently.
- **The crash must survive the release build.** The hardware runs `-O3 -DNDEBUG`. `assert()` is therefore the *wrong* tool ŌĆö it's stripped, and historically pulled in newlib's `__assert_func ŌåÆ fprintf`. The right tool is the project's existing `HS_CHECK(c)` (a `__builtin_trap()` on a predicted-not-taken cold-path branch, zero hot-loop cost).
- **Bounded/soft handling is correct only for genuine *transient* conditions** ŌĆö a DMA overrun dropping a frame, a missed sync pulse, an unbound particle pool early-returning. Distinguish "should never happen" (trap) from "happens occasionally by design" (degrade).

The engine *mostly* gets this right: `Arena::allocate`, `ArenaVector::push_back/append_bulk`, the `vertex_orbit` anti-hang trap, and the plot `_steps_cache` guard are all proper `HS_CHECK` traps. The remaining gaps ŌĆö pixel-sink bounds, in-orbit `assert`s, `Canvas::operator()` bounds, math-core silent fallbacks ŌĆö are where invariant violations still slip past the device build, and they are the spine of the P0/P1 list below.

1. **Ō£ģ FIXED (2026-06-05) ŌĆö debug-only precondition tripwire, zero device cost.** Traced every producer feeding the terminal sink before changing anything: the 3D path full-wraps via `vector_to_pixel` (`geometry.h:281` uses the safe `wrap()`), and `AntiAlias`/`Blur`/`ChromaticShift` keep x within `[-1, W]` before forwarding (`filter.h` AntiAlias `x0/x1` come from `modf` of an already-wrapped coord; Blur/ChromaticShift use full `wrap()`). So `fast_wrap`'s `[-W, 2W)` precondition **holds for every finite coordinate** ŌĆö the OOB is unreachable in practice and the only real trigger is a non-finite (`NaN`) coord reaching the float sink, where `static_cast<int>(round(NaN))` is UB. Because `fast_wrap` is on the hottest path of all, an always-on `HS_CHECK` branch was rejected (per the perf constraint). Instead the precondition is now enforced **debug-only** (`assert`) at both terminal sinks (`filter.h` int sink: `assert(x >= -W && x < 2*W)`; float sink: `assert(std::isfinite(x) && std::isfinite(y))` + range): stripped under `NDEBUG` on device (literally zero hot-loop cost), but fires in the native test suite and WASM-debug build ŌĆö exactly where a NaN-producing effect or a future out-of-range filter is first exercised. **Verified:** full native suite (every effect rendered at 288├Ś144 with asserts on, plus all filter/plot/scan tests) passes with the asserts active, confirming the invariant holds and the tripwire is non-spurious. The genuinely correct place to stop a `NaN` is its source (an effect's 0/0), which this tripwire will surface; the per-pixel sink stays lean. Original finding below.

### P1 ŌĆö Wrong output, latent crash, or fail-fast gap

2. **Ō£ģ FIXED (2026-06-05).** Two 16-bitŌåÆ8-bit color truncations in shipping effects, both fixed by widening the cast to `uint16_t` (the channel-wise lerp/scale stays in `[0, 65535]`, so this is behavior-preserving; the separate rounding-bias item is tracked at P2 #9):
   - `Voronoi.h:90-95` ŌĆö smoothed cell color was `static_cast<uint8_t>(...)` on `uint16_t` linear channels (0ŌĆō65535). With ~200 sites the smoothing path runs on essentially every pixel, so colors wrapped mod-256 ŌåÆ mostly-dark garbage. Now `static_cast<uint16_t>`.
   - `SphericalHarmonics.h:167` ŌĆö negative-lobe green channel was force-cast to `uint8_t`, truncating up to 65535. Now `static_cast<uint16_t>(pos.color.g * 0.8f)`.
   - **Verified:** native suite passes (both effects render in the 288├Ś144 smoke harness).

3. **Three effects abort at 288├Ś144 (quarantined: SplineFlow, TestShapes, Thrusters).** All trip *fixed-capacity* guards in the plot path at full resolution: SplineFlow overflows its `MAX_TRAILS`/trail buffer with closed-geodesic sample fragments; TestShapes overflows `_steps_cache` (`plot.h:192`); Thrusters faults in the `DistortedRing`/`Ring` plot path. These are correct fail-fast traps, but they make three shipping effects unrenderable standalone at Phantasm resolution. **Fix:** size the plot caches/trail buffers off `W` (as the Face dist-LUT already does, commit `f0ac2ea`) or clamp sample counts; then remove from the quarantine list. **Confirm whether they also reproduce on-device** ŌĆö under the failure philosophy these are live-show crashes waiting to happen.

4. **Ō£ģ FIXED (2026-06-05).** WASM `setEffect` format-string defect. `wasm.cpp:154-156` built a string with `snprintf` then passed it as the *format* argument to `hs::log` (printf-style ŌĆö `vprintf` on host, `vsnprintf` on device, `platform.h:52,286`); an effect name containing `%` made it read nonexistent varargs ŌåÆ UB/crash. Now `hs::log("WASM: setEffect called with %s", name.c_str());` with the intermediate buffer dropped, matching every other call site. (WASM-only TU; not exercised by the native suite, and the Emscripten toolchain isn't provisioned in the review environment, so this was not compile-verified here ŌĆö but the change is the canonical printf-style form.)

5. **Ō£ģ RECLASSIFIED (by-design) + HARDENED (2026-06-05).** Originally filed as a P1 dangling-reference bug; on review it is **not a live bug** and the borrow model is **correct by design**, not a smell. `Motion` stores an `Fn<>` capturing `&path_obj`, and `MeshMorph` stores two non-owning `FunctionRef`s ŌĆö both deliberately *borrow* effect-owned state that outlives the `Timeline`. This is load-bearing: animations live in `Timeline`'s fixed **inline-storage budget** (`MAX_ANIM_SIZE`) with no heap/arena allocation, and `FunctionRef` is intentionally an 8-byte non-owning callable (`concepts.h:32` ŌĆö `ctx_`+`thunk_`) for exactly this reason. The agent's suggested fix (make the callbacks *owning*, e.g. `Sprite::SpriteFn`) is therefore **rejected** ŌĆö an owning `Fn<ŌĆ”,N>` would inflate the animation's inline size against the timeline's constrained RAM, defeating the whole point of `FunctionRef`. All four real call sites already pass effect-owned lvalue members (`ChaoticStrings`/`Comets` `ProceduralPath path;`; `MeshFeedback`/`HankinSolids` owning `Fn<>` `draw_morph_*_fn_`). The only genuine weakness was that the borrow was safe-by-*convention*: the constructors also silently bound to temporaries. **Hardening applied:** a compile-time rvalue guard (`= delete` overload constrained by `!std::is_lvalue_reference_v<ŌĆ”>`) on both `Motion` and `MeshMorph` constructors makes passing a temporary (e.g. an inline lambda or `ProceduralPath{...}`) a **compile error** instead of a silent dangle ŌĆö non-owning footprint preserved, zero runtime cost. The guard was deliberately placed on these *storing* constructors, **not** on `FunctionRef` itself (which is legitimately constructed from temporaries on the synchronous hot path, e.g. shaders into `Scan::draw`). **Verified:** native suite passes (all four call sites still resolve to the primary ctor), and the borrow contract is now locked by permanent `std::is_constructible` `static_assert`s in `tests/test_animation.h` (`borrow_guard`) that confirm lvalues are accepted and temporaries rejected in both types (including the mixed-slot case) ŌĆö these fail to compile if a refactor turns the SFINAE guard into a no-op.

6. **`MindSplatter` over-subscribes the 335 KB device arena.** Its 1024-particle system + baked palette + pools only fit because `HS_TEST_BUILD` widens the arena to 8 MB; on-device it is a hard arena-overflow fault. **Fix:** reduce particle count / trail history under the device budget.

7. **Ō£ģ FIXED + RESOLVED PER-SITE (2026-06-05).** The blanket "promote every invariant `assert` to `HS_CHECK`" framing was wrong ŌĆö the three cited site-classes have different characters, so each was resolved on its merits:
   - **Mesh-orbit buffer guards ŌåÆ promoted to `HS_CHECK`.** `conway.h` (5 sites: the `orbit_buf[orbit_count]` guards inside `vertex_orbit` callbacks) and `hankin.h:193,200` (the hand-rolled `compile_hankin` orbit writing `face_indices[count]`) were stripped-on-device `assert`s guarding fixed-capacity writes. These are **cold mesh-compile paths** (run per-vertex during a shape transition, not per-pixel), so an always-on trap matches `vertex_orbit`'s existing posture (`conway.h:90`) with negligible cost. A corrupt/non-manifold twin graph now hard-faults at the overflow site instead of silently writing past the buffer.
   - **`relax` hand-rolled orbit ŌåÆ added a missing anti-hang `HS_CHECK`.** The `MeshOps::relax` spring loop (`conway.h:~801`) walks the half-edge orbit by hand and accumulates a force vector ŌĆö so it had *no* buffer assert, but also *no* iteration bound (unlike `vertex_orbit`). On a corrupt twin graph it could spin forever and hang the device. Added an `orbit_count < halfEdges.size()` trap mirroring `vertex_orbit`.
   - **`Canvas::operator()` bounds (`canvas.h:317,329,340`) ŌåÆ kept debug-only, by design.** This is the **per-pixel write accessor ŌĆö the hottest path of all**. The project's own failure philosophy says traps go on cold paths *"never the per-pixel loop"*, and P0 #1 already added the upstream precondition tripwire that keeps this accessor lean. Promoting it would contradict both.
   - **Math-core `Vector::normalize` / `Quaternion::normalize` / `angle_between` (`3dmath.h:198,398,715`) ŌåÆ kept graceful fallback, by design.** A zero-length normalize / zero-vector angle is **legitimate transient degenerate geometry** (parallel cross-products at poles, coincident vertices), not an invariant violation ŌĆö and the non-NaN fallback is *load-bearing* (effect code relies on `normalized()` returning a valid axis for parallel crosses instead of propagating NaN). Per the project's explicit "transient ŌåÆ degrade, not trap" rule, hard-faulting here would crash live shows on benign frames. Left as-is. (The per-call `hs::log` on these hot paths is a minor spam/perf wart, tracked with the broader per-frame-logging cleanup, not converted to a trap.)
   - **Verified:** native suite passes (all Conway operators, Hankin patterns, and the solids registry exercise the promoted guards without tripping).

8. **Ō£ģ ALREADY RESOLVED in daydream (verified 2026-06-05; commit `f8e6aa6`).** The finding over-stated the bug. Tracing the current `applyResolution` confirms the effect **is** always re-sent after `workerSetResolution`: when the current effect stays valid, line 564 calls `applyEffect(preserveParams)` ŌåÆ `workerSetEffect`; when it changes, `appState.set('effect', ŌĆ”)` synchronously fires the subscriber ŌåÆ `applyEffect()` ŌåÆ `workerSetEffect`. The only genuine residual was that the worker's `setResolution` handler left the clip un-applied, relying on that external ordering; daydream commit `f8e6aa6` made the handler self-contained by calling `applyClip()` directly after `engine.setResolution` (no-op while no effect is bound). No Holosphere-side change; this entry was stale because the fix lived in the sibling daydream repo. (Several other daydream-side findings ŌĆö #6, P2 #10/#12 ŌĆö were likewise addressed in that repo: commits `8ce3693`, `1c3778d`, `9c69437`.)

## Overall impression ŌĆö technical and artistic merit

**Technical merit: very high.** This is the work of an engineer who understands their constraints to the microsecond and has built an architecture that earns its complexity. The compile-time `<W,H>` specialization is the right call for embedded firmware; the 16-bit linear color pipeline is correct where most LED-art codebases are casually wrong; the partitioned-arena memory model with explicit `Arena&` threading is disciplined functional-style memory management rarely seen outside systems code; and the variadic filter pipeline that automatically lifts 2DŌåö3D coordinates at compile time is a genuinely elegant piece of template design. The fail-fast philosophy is not just stated but *mostly implemented* ŌĆö `Arena::allocate` trapping at the violation site is what makes dozens of "unchecked" downstream derefs provably safe. The reaction-diffusion-on-a-Fibonacci-lattice work, the Hopf fibration, the Hankin-method Islamic star patterns generated live from Conway-operated polyhedra, and the SDF volumetric raymarcher are all serious computational-geometry achievements running in real time on a 600 MHz microcontroller.

The codebase's weaknesses are the *normal* weaknesses of an ambitious solo project: a handful of places where the discipline lapsed (`assert` where `HS_CHECK` was meant, two stray 8-bit truncations, dangling references from convenience-captured temporaries), some copy-paste that wants extracting, and a test suite whose harness quality (real algebraic invariants) outruns its coverage (effects, integration, and the bridge are thin). None of these are architectural; they are the punch-list of a strong system that needs a hardening pass.

**Artistic merit: exceptional for the medium.** A persistence-of-vision LED sphere is an unusually demanding canvas ŌĆö the geometry is spherical, the time budget is brutal, and the output is physical light, not pixels behind glass. The effect library is not a collection of demos but a coherent aesthetic: mathematically-derived motion (M├Čbius warps, spherical harmonics, fibrations), perceptually-correct color in OKLCH, and motion blur derived for free from the orientation history. The decision to render in linear light and blend in 16-bit is an *artistic* choice as much as a technical one ŌĆö it's why the gradients and multi-layer composites look clean rather than muddy. The web simulator compiling the identical C++ to WASM so the art can be previewed and shared without the hardware is the kind of end-to-end thinking that elevates the whole project.

**Bottom line:** an **AŌłÆ / B+** codebase ŌĆö top-decile engineering and artistry for a solo embedded-art project, with a clear, bounded, and mostly-mechanical path to an unqualified A. Land the P0 sink hardening and the P1 correctness/fail-fast cluster, and this is an A-grade system end to end.
