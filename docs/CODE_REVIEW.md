# Holosphere / Daydream — Code Review

**Date:** 2026-06-30
**Scope:** The full Holosphere C++ rendering engine (`core/`, `effects/`, `hardware/`, `targets/wasm/`, `targets/Phantasm/`, `tests/`) and the daydream web simulator (`*.js`, `tools/*.js`). Out of scope by request: `core/effects_legacy.h`, `targets/Holosphere/Holosphere.ino`, `core/rotate.h`. Generated artifacts (`holosphere_wasm.js`), vendored `three.js/`, and third-party `FastNoiseLite.h` were excluded from defect scoring.

## Methodology

The codebase (~44k LOC C++, ~4.8k LOC JS) was partitioned across **14 independent component reviewers**. Each reviewer consulted the README to place its files in the architecture, audited every file in its slice across all software-quality dimensions, and — critically — **dispatched a fresh, independent sub-agent to validate every candidate finding** before inclusion. A finding was admitted only if an independent reader confirmed it was real, correctly reasoned, and not already handled elsewhere.

This validation layer was decisive. Roughly thirty plausible-looking candidates were **refuted** under independent scrutiny (correct-by-design, already-guarded, or unreachable), and in one instance a reviewer's own dismissal of a hardware trap was **overturned** by its validator and escalated to the single High-severity finding below. The low final defect count is therefore a signal of genuine maturity, not shallow review: four entire subsystems — core math/color, the mesh system, the WASM bridge + native test suite, and the geometry tools — returned **zero** validated defects after exhaustive probing.

## Quality Dimension Grades

| Dimension | Grade | Rationale |
|---|---|---|
| **Correctness & Numerical Robustness** | A− | Degenerate paths (poles, antipodes, div-by-zero, NaN/Inf clamps) are guarded essentially everywhere; fixed-point conversions clamp-and-round uniformly. Pulled down only by one reachable hardware timing trap (#1) and a localized animation teleport (#2). |
| **Memory Safety** | A | Arena partitioning is alignment-correct and wrap-proof; every per-effect persistent+scratch footprint is `static_assert`-pinned to the arena budget; container growth and index narrowing trap at cold seams. No memory-safety defect found across the whole audit. |
| **Concurrency / ISR Safety** | A− | The triple-index double-buffer uses relaxed atomics with a rigorously specified single-writer/single-observer contract, correct for the Cortex-M7. The one blemish is a sync-protocol timing margin that lets an always-on device trap fire under a masked-ISR coast (#1). |
| **API Design & Interface Expressiveness** | A− | The variadic filter pipeline, the `Fragment`/shader interface, explicit `Arena&` threading, and Style-driven feedback are clean and composable. Minor: the JS `export()` conflates value streams with `Params` aggregates (#3), one filter leaks a public member (#14), and a param/doc inconsistency (#9). |
| **Architectural Elegance** | A | Compile-time `<W,H>` specialization, the three-domain (world/screen/pixel) pipeline, the pure-core / device-shell split that makes hardware logic host-testable, X-macro single-source rosters, and CRTP effect bases are coherent and consistently applied. |
| **Performance** | A | Hot loops avoid virtual dispatch and allocation; fast-trig with documented error bounds, baked palette LUTs, per-face adaptive LUTs, cold-path-only traps. One per-fragment `sinf` deviates from the `fast_sinf` convention (#10). |
| **Error Handling & Fail-Fast Discipline** | A− | `HS_CHECK` is principled (cold paths only, survives `NDEBUG`), the death harness verifies traps actually fire, and the JS layer reject-and-returns at every untrusted boundary. The reachable beacon trap (#1) is the one place a fail-fast guard can fire during normal operation. |
| **Maintainability & Readability** | A− | Terse, fact-focused comments; load-bearing doxygen. A few footguns remain: a shadowed opposite-signed variable (#16), an invariant enforced only by statement ordering (#15), and one inert dead literal (#17). |
| **Naming & Style** | A | House conventions (Filter namespace placement, no leading-underscore JS privates, no finding-refs in comments) are honored consistently across both languages. |
| **Documentation (comments + README)** | A− | Code comments are accurate and the README is unusually thorough and precise. Several drift items keep it from an A: an undocumented slider (#9), an overstated recording guarantee (#11), and the `export()`/§10.6 mismatch (#3). |
| **Test Coverage & Quality** | B+ | Excellent *infrastructure*: a 37-case death harness with exact trap-shape probing, X-macro module rosters cross-checked against CMake both directions, WASM↔JS parity suites, and host-testable hardware cores. But there are real *coverage* gaps: the Effect single-live trap (#15), several animation APIs (#6), the sync beacon margin (#1), the SmoothUnion near-pole path (#4), and DOM-level keyboard selection (#13). |

**Overall: A−.** This is a mature, internally consistent, and unusually well-hardened codebase. The defects below are dominated by Low-severity polish, documentation drift, and test-coverage gaps; the single High finding is a narrow but genuine reachable device trap, and the two Medium findings are localized behavioral bugs with clear fixes.

## Prioritized Fix List

Findings are numbered sequentially across all priority tiers. Each is independently fixable as one focused change (one finding → one commit) per the code-review-fix workflow.

### P1 — High

1. **Beacon symbol can overrun the HALF boundary and trap the master in the field.** `hardware/pov_sync.h:980` (`SymbolEmitter::schedule_boundary` trap) + `maybe_schedule_beacon` (`hardware/pov_sync.h:1530-1559`). The protective comment at `pov_sync.h:1539-1540` assumes the beacon starts at ≈column 72 (a 72-column margin to HALF), but the beacon start is unbounded in `[W/4, W/2) = [72, 144)` and its worst-case span is exactly **55 columns**, so the true slack is only **17 columns** (start must be ≤ column 89). A masked-ISR coast of ~18 columns (well inside the `kMinSafeHalfRevs=16` overflow-safe window, so `position()`'s own overflow assert does *not* pre-empt it) that lands the master at columns 90–143 with `boundary_` still ZERO queues the beacon there; its tail (columns 145–198) overruns HALF at column 144, and the subsequent on-time HALF crossing trips `HS_CHECK(pulses_left_ == 0 && queue_pos_ >= queue_len_)` → `__builtin_trap` on the master board. This is a designed-for coast regime, not an impossible one. *Note:* this overturns the same trap's earlier "untested but unreachable" framing — it is reachable. **Fix:** bound the beacon start in `maybe_schedule_beacon` (skip/early-return when `position(now) > 89`, i.e. `W/4 + 17`), mirroring the boundary symbol's own late-censor at `pov_sync.h:977-979`; add a host regression test in `tests/test_pov_sync.h` driving a worst-case beacon-then-HALF coast and asserting no trap. Confirm against the real pitch/gap/oversample constants before landing.

### P2 — Medium

2. ✅ **`Motion::set_duration` teleports the orientation by one frame (missing reanchor).** `core/animation_motion.h:95`. `Motion::step` integrates the path as relative deltas off the carried `prev_frame_` baseline, sampling `path_frame(sub_t / duration)`. `set_duration` mutates `duration` in place without clearing `have_prev_frame_`, so the next step's first delta `frame_new · frame_old⁻¹` snaps the bound `Orientation` by the gap between the old and new parameterizations — exactly the discontinuity `reanchor()` (`animation_motion.h:112`) exists to absorb for path-function swaps. A live duration slider (the documented use case) therefore produces a visible head jump. **Fix:** have `set_duration` reset `have_prev_frame_ = false` (as `reanchor()` does) when the new duration differs, so the next step re-seeds the baseline and flows incrementally.

3. **The simulator's `export()` emits readonly and non-float params, producing wrong or non-compiling C++.** `daydream.js:275-279`. The loop emits a `toFixed(4)` value for *every* registered param in registration order, including **readonly** ones and ignoring non-float aggregate members. Hand-authored `Presets<>` tables deliberately omit readonly/engine-written fields and interleave non-floats. Two confirmed breakages: (a) `MindSplatter` registers a readonly `active_count` (`effects/MindSplatter.h:68`) that the preset omits — export bakes a live per-frame particle count into the preset; (b) `DreamBalls::Params` (`effects/DreamBalls.h:34-42`) interleaves `const char* solid_name` and `const Palette* palette` among the floats, so export yields a 5-float list that neither matches the aggregate's arity nor its positions and won't compile. README §10.6's "suitable for `Presets<…>` arrays" holds only for pure-float effects. **Fix:** filter on `params[i].readonly`, emit a name-keyed designated-initializer form (or scope the feature to pure-float effects and say so in §10.6), and add a test pinning export output for a representative effect.

### P3 — Low

4. **`SmoothUnion` scan-interval horizontal pad omits the `1/sin(φ)` factor, clipping near-pole welds.** `core/sdf.h:1020` (pad applied at `sdf.h:1024` and `sdf.h:1029`). The smooth weld pulls the surface outward by up to `k` radians, but the column pad is `k·W/(2π)` — the equatorial angle→column conversion with no `1/sin(φ)` division that every leaf emitter uses (`annular_band_angles` at `sdf.h:446-447`, `emit_cap_interval` at `sdf.h:369`). At `sin(φ)=0.1` the pad under-covers by ~10×, leaving an unshaded sliver, because `scan_region` only shades columns inside an emitted interval. The vertical pad is correctly uniform; only the horizontal axis needs the latitude factor. Equatorial welds (the common case) are unaffected, and the path is untested (existing SmoothUnion tests exercise only `distance()`). **Fix:** scale the pad per-row by `1/max(sin φ, ε)` clamped to ≤ W, or fall back to a full-row scan when the near-pole pad would exceed full width; add a high-latitude weld-coverage regression test.

5. **`deep_tween` drops an interior length-1 frame's age slice (latent robustness gap).** `core/animation_timeline.h:536-547`. An interior frame with `length()==1` takes `start_j=1`, so its inner loop emits nothing, yet `span = last+1` still reserves its `global_t` slice — leaving a ~`1/span` hole in the motion-blur age ramp. *This is currently unreachable:* all three consumers (Comets/ChaoticStrings/RingSpin) drive via `Motion` (always upsamples to ≥2) or fixed-speed `RandomWalk` (delta ≫ TOLERANCE), so an interior length-1 frame never enters a trail. It is a robustness/documentation gap in a generic `Tweenable` helper, not a live rendering defect. **Fix:** either emit the lone sub-position of an interior length-1 frame or exclude motionless interior frames from `span`; document the contract; add a moving/motionless/moving `deep_tween` test.

6. **Several public animation APIs have zero behavioral test coverage.** `tests/test_animation.h` (dispatch at `run_animation_tests`). Absent from the suite: `MobiusFlow::step` (only borrow-contract `static_assert`s), `RandomTimer` (only `PeriodicTimer` is exercised), `PeriodicTimer::set_period`, `ParticleSystem` emitter dispatch (`add_emitter`), and `Motion::set_duration`/`reanchor` — the last directly hiding finding #2. **Fix:** add behavioral tests for each (e.g. `RandomTimer` fires within `[min,max]`; `set_period` reschedules from now; a `set_duration`-then-step continuity check) and wire them into `run_animation_tests` per the project's "new tests into CI" rule.

7. **The render animation loop is never stopped on teardown.** `daydream.js:745` / `driver.js` `dispose()` (`driver.js:836-869`). `renderer.setAnimationLoop(...)` is installed but never cleared — neither `disposeApp()` nor `Daydream.dispose()` calls `setAnimationLoop(null)`. On a real `pagehide` discard, `dispose()` nulls `dotMesh` and disposes the renderer while the rAF callback keeps firing, dereferencing the null `dotMesh.instanceColor` and throwing every frame until unload — contradicting `dispose()`'s stated "no live observer firing into a dead scene" contract. (bfcache freeze is correctly unaffected.) **Fix:** call `this.renderer.setAnimationLoop(null)` in `Daydream.dispose()` before disposing the renderer, or expose a `stop()` the host calls first.

8. **`export()` uses raw `toFixed(4)` instead of the authoritative `formatFloatCpp`.** `daydream.js:269-274`. Every other C++-emitting path (lissajous, palette, solid, spline codegen) routes through `formatFloatCpp` (`tools/cpp_format.js`, pinned by `cpp_format.test.js`), but `export()` rolls its own formatter, so values with magnitude `< 0.00005` collapse to `0.0000f`, losing a meaningful small significand the other emitters preserve. **Fix:** reuse `formatFloatCpp` in `export()` so all C++ emission shares one tested formatter.

9. **PetalFlow's `Density` slider is registered but undocumented.** `README.md:1620` (and `effects/PetalFlow.h:63`). `init()` registers four live sliders — Twist, Speed, Alpha, **Density** (range `[0.5, 2.5]`, default 1.0, wired through `spacing()` at `PetalFlow.h:178`) — but README §9 lists only "Twist, Speed, Alpha." Doc-vs-code drift that can mask a future accidental param change. **Fix:** update the README §9 PetalFlow line to include Density.

10. **Per-fragment `sinf()` in the MobiusGrid longitude shader deviates from the `fast_sinf` hot-path convention.** `effects/MobiusGrid.h:249`. The `draw_longitudes` shade lambda runs per rasterized fragment and uses libm `sinf` where the engine uses `fast_sinf` on hot paths. (The neighboring `sqrtf`/`logf` have no fast equivalent and the per-curve `cosf`/`sinf` run only ~20×/frame, so leave those.) Small magnitude — a consistency cleanup, not a material lever. **Fix:** replace `sinf(...)` at `MobiusGrid.h:249` with `fast_sinf(...)`.

11. **The recording guarantee in README §10.9 is stated unconditionally.** `README.md` §10.9. The README claims recorded video is "locked to the effect's simulation rate … regardless of how fast the browser actually renders" with "byte-perfect repeatability," but the code's own timed-fallback path (`recorder.js:131-149`, used when `track.requestFrame` is absent) samples on the wall clock — a caveat `recorder.js`'s own `elapsedFormatted` JSDoc (`recorder.js:257-261`) already acknowledges. **Fix:** add a one-sentence caveat to §10.9 noting the timed fallback on browsers lacking `track.requestFrame`.

12. **`gui.js` is missing the copyright/license header every sibling module carries.** `gui.js:1`. Every other in-scope module opens with the `Copyright 2025 Gabriel Levy … Polyform Noncommercial License 1.0.0` block; `gui.js` begins with a blank line then `import`. Licensing inconsistency. **Fix:** prepend the standard header block.

13. **No DOM-level test for keyboard selection in the sidebar.** `tests/sidebar_logic.test.js`. The suite exercises only pure helpers (`navTargetIndex`, `sortItems`); `EffectSidebar.onKeyDown`'s Enter/Space selection path is never driven against real button elements. The behavior is correct (keydown `preventDefault` at `sidebar.js:279` suppresses the native click so `onSelect` fires once), but a regression dropping `preventDefault` (reintroducing double-selection) would not be caught. **Fix:** add a jsdom-backed test dispatching Enter and Space on a focused option button asserting `onSelect` is called exactly once; wire it into the existing runner.

14. **`VertexReplicate::rotations` is a public data member.** `core/filter.h:641`. `std::array<Quaternion, N> rotations;` is public, unlike every other filter in the file (`Orient`, `OrientSlice`, `Hole`, `Replicate`, `Mobius`, `Trails`, `Feedback`, …), which keep state private. Callers can mutate the precomputed rotation table out from under the filter, breaking encapsulation consistency in the headline composability layer. It is only written in the constructor and read in `plot()`. **Fix:** move it into a `private:` section.

15. **The single-live-`Effect` `s_alive` trap has no death test.** `core/canvas.h:47` (guard) / `tests/test_death.h` (missing case). The `Effect` constructor's `HS_CHECK(!s_alive, …)` protects the shared static framebuffers and double-buffer indices, but unlike its structural twin (single-live `Timeline`, covered by `case_timeline_double_construct`), no `effect_double_construct` death case exists. README:126 explicitly claims the harness verifies "these traps," so this one is asserted-verified but actually unverified — and the invariant is enforced only by statement ordering at each effect-switch site (the `.reset()`/`delete` preceding the rebuild), which a future refactor collapsing it into a single assignment would silently break. **Fix:** add a death case constructing two live effects so the second ctor trips `s_alive`; register it in `all_cases()` and the module/X() list.

16. **Raymarch's local `view_dir` shadows the opposite-signed shader parameter of the same name.** `effects/Raymarch.h:157`. `view_dir = -vertex` is the ray-travel direction (camera→scene) passed to `Scan::Volume::draw`, while `shadeBlinnPhong(..., view_dir, view_dir, ...)` at `Raymarch.h:167` is correctly passed `vertex` (toward-viewer) — the opposite sign. Both being named `view_dir` makes the correct call read as a sign bug and invites a "fix" that would break Fresnel/specular. No runtime defect (confirmed correct). **Fix:** rename the local to `ray_dir`, or add a one-line comment at the call site noting the deliberate sign flip.

### P4 — Very Low / Informational

17. **Inert `base_scale` literal in ChaoticStrings duplicates a param default.** `effects/ChaoticStrings.h:184`. `ScaleModifier scale_mod{200.0f, &params.scaleFactor};` passes `200.0f` as `base_scale`, but since the bound pointer is always non-null, `ScaleModifier::modify()`'s ternary never reads it — the literal is dead and only coincidentally mirrors `params.scaleFactor`'s default. Per the repo's "remove dead code over documenting" convention, **Fix:** drop it to the neutral default (`{1.0f, &params.scaleFactor}`) so the slot's irrelevance is obvious. Below the bar for action on its own.

18. **(Informational — no fix required) Documented DMA wedge-detection blind spot.** `hardware/dma_led.h:145-152`. `checkStaleTransfer()` runs only from `submitFrame()`'s overrun-drop path, so a DMA channel that wedges during a sustained dark-latched window (`POVSegmented` stops submitting once `dark_latched_`, `pov_segmented.h:577`) is never surfaced — the dark-strip symptom is indistinguishable from an intended dark frame. This is already called out in the header comment and is design-acknowledged. Flagged only for audit visibility; if field telemetry ever shows wedges during dark windows, a foreground periodic `checkStaleTransfer()` poll in `run_show`'s loop would close it.

## Clean Subsystems (zero validated defects)

After exhaustive probing with independent validation, the following returned no admissible findings:

- **Core math & color** (`3dmath.h`, `geometry.h`, `color.h`, `color_luts.h`, `palettes.h`, `util.h`, `easing.h`, `waves.h`) — every degenerate trig/color path is clamped, indexed in-bounds, and test-covered.
- **Mesh system** (`mesh.h`, `conway.h`, `hankin.h`, `solids.h`, `spatial.h`, `static_circular_buffer.h`, `reaction_graph.*`) — half-edge invariants, Conway operator capacity/winding, arena polarity, and KDTree bounds are all correct-by-design or guarded, with Euler/manifold/tie-boundary tests.
- **WASM bridge & native test suite** (`targets/wasm/*`, `targets/Phantasm/Phantasm.ino`, `tests/`) — single-sourced marshaling and resolution rosters, transactional teardown, self-pinning module lists, exact-trap-shape death harness.
- **daydream geometry tools** (`tools/*.js`) — Möbius/Lissajous/spline/palette math is unit-correct and C++↔JS parity is WASM-tested; codegen rejects invalid tokens before emission.
