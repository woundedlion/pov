# Holosphere — Code Quality Review

*Fourth independent multi-agent audit, 2026-06-10. Fifteen reviewers each read one component end to end against the 1,900-line README, recovering the intended contract before judging the code; every candidate finding was then handed to a separate adversarial validator that re-opened the source and weighed it for **truth, relevance, and whether a fix is worth a developer's time** before it could enter this report. Of sixteen candidates raised, eleven survived and five were dismissed. The grades, the synthesis, and the appraisal in §§5–7 are formed independently and deliberately do **not** reuse the prior audits' framing, organizing questions, or conclusions. Out of scope by request: `core/effects_legacy.h`, `targets/Holosphere/Holosphere.ino`, and `core/rotate.h`; `core/FastNoiseLite.h` is treated as vendored. Letter grades (A+ … F) are measured against a high professional / graphics-middleware bar — not "does it run."*

*This pass post-dates three prior rounds whose findings were all marked remediated. Those fixes were not re-litigated, and the previously dismissed candidates (the dead-code orbit guard, the truncate-vs-round non-issue, the by-design list in the prior §4a) were not re-raised. What is new here is that this pass, unlike the last, found its surviving debt **inside live code paths** rather than only at unreached edges — five of the eleven are reachable today, and two are genuine logic/visual defects rather than doc drift.*

---

## 1. Executive Summary

Holosphere is a single C++20 spherical rendering engine (~50k LOC across `core/`, `effects/`, `hardware/`, `targets/`, plus the sibling `daydream/` JavaScript simulator) that compiles to two Teensy 4.x persistence-of-vision LED-sphere firmware targets — a 96×20 single-controller *Holosphere* and a four-microcontroller 288×144 *Phantasm* — and to a WebAssembly module driving a Three.js browser twin. One set of effect classes, one 335 KB arena, and one per-frame `Pixel16[]` buffer run unchanged across all three targets; resolution is a compile-time `<W,H>` template parameter, so the device build pays nothing for generality.

The engine's working core remains as strong as three prior audits found it. **Eight of the fifteen components returned no surviving findings at all** — memory, the filter/animation pipeline, the field/shader effects, the particle/curve effects, the WASM bridge's untrusted boundaries, the math core, the hardware drivers, and the daydream web layer each came back clean after validation. The abstractions the prior passes singled out (the compile-time variadic `Pipeline<W,H,Filters…>`, the lock-free triple-index double buffer, the OKLCH 16-bit-linear color path, the verified fail-fast death harness) were re-examined by fresh readers and held.

What distinguishes *this* pass is the **character** of the eleven that survived, not their count. The prior audit's residue was almost entirely doc and consistency notes at code the shipped product never reaches. This pass reached deeper and found a thin but real scatter of defects in *executing* code:

- a **genuine logic bug** — `BZReactionDiffusion`'s stochastic perturbation picks a species with `rand_int(0, 2)`, which on the project's own half-open convention can only return `{0, 1}`, so species C is *never* perturbed and one arm of a three-way ternary is dead code (effects/BZReactionDiffusion.h:141);
- a **reachable latent trap** — `classify_faces_by_topology` hashes `count` corner angles for every face but only *computes* them when `count >= 3`, so a degenerate face (which the WASM `classifyFaces()` path can be fed via `fromData()`) reads uninitialized scratch and produces a non-deterministic topology id instead of the project's mandated trap (core/mesh.h:404-422);
- a **silent visual-correctness bug** — `Plot::Flower` centers its planar projection basis on the opposite hemisphere from where its petals actually sit, so for the common small-radius case the intended tangent-plane projection never runs and the flower silently degrades to a geodesic polygon (core/plot.h:1279);
- an **inert user control** — Dynamo's "Trail Len" slider writes a value that is read exactly once at construction and never again, so dragging it does nothing, and an adjacent comment affirmatively claims a runtime control that does not exist (effects/Dynamo.h:40, :204-205);
- a **partial safety guarantee** — `Pixel16::operator*` clamps with `std::clamp` (which passes NaN through) before a `uint16_t` cast, reintroducing exactly the NaN→float→int UB the project deliberately hardened against in the sibling `blend_alpha` (core/color.h:85-87).

The rest are P3 cleanups: a near-twin `to_short` that clamps in the wrong domain, a redundant `std::round` over an already-integer RNG result, a `Presets<…,0>` accessor that is UB while its own `next()/prev()` guard against the empty case, a WASM log line that passes a runtime buffer as a printf format string against the file's own stated rule, and a death-test header index that omits the Plot seam it actually exercises.

**No P0 and no P1 survived.** None of the eleven mis-paints shipped hardware, and none is a memory-safety defect reachable on the registered inputs the product renders. But the texture has shifted: where the last pass could say the debt lived only at the boundary seams, this one found correctness debt — narrow, low-blast-radius, but real — in the engine's own effects and primitives. **Net: A− work.** A cohesive, exceptionally documented and verified engine whose surviving defects are small and individually cheap to fix, but which this deeper pass shows are *not yet* confined to the unreached edges.

## 3. Cross-Cutting Quality Dimensions

*Whole-product synthesis across all fifteen components, after validation.*

### Correctness & Robustness — **A−**
No memory-safety defect (use-after-free, OOB, alignment, arena overflow) was confirmed reachable on the inputs the product renders, and the arena bounds math remains provably wrap-safe. But this pass found more substantive correctness debt than the last, all of it narrow: a dead-ternary-arm logic bug starving `BZReactionDiffusion` species C of perturbation (BZReactionDiffusion.h:141); a half-applied guard in `classify_faces_by_topology` that hashes uninitialized scratch for degenerate faces — *reachable* via the WASM `classifyFaces()`/`fromData()` tool path, which bounds face vertex counts above but not below (mesh.h:404-422); a wrong-hemisphere projection center silently defeating `Plot::Flower`'s planar strategy (plot.h:1279); and a partial NaN-safety guarantee in the central `Pixel16::operator*` (color.h:85-87). None mis-paints shipped hardware — the Flower has an antipode fallback, the BZ field still animates, the mesh path is unreached by registered solids — but three of the four are in live code, which is why the band sits at the low end of A−.

### Architecture & Design Elegance — **A**
Untouched by this pass. The compile-time `Pipeline<W,H,Filters…>` with automatic world/screen/pixel bridging via discarded `if constexpr` branches, the terminal-filter `static_assert` that turns a mis-wired pipeline into a *compile* error, the one-engine/three-targets seam that leaks nothing into effect code, the owned-vs-borrowed `MeshState` duality, the purely-functional Conway operators over ping-ponged arenas, the CRTP `ReactionDiffusionBase`, and the self-registering `REGISTER_EFFECT` factory are all cohesive and were re-affirmed by independent readers. No structural change is indicated anywhere.

### Interface Expressiveness — **A−**
The `registerParam` → WASM → auto-generated GUI reflection seam, the documented per-rasterizer `Fragment` register conventions, `SolidBuilder`'s fluent Conway chaining, and `.then()` animation sequencing remain expressive and hard to misuse. Docked this pass for two concrete seams a contributor *does* trip over rather than merely *could*: Dynamo exposes a "Trail Len" slider whose value is captured once at construction and never re-read, so the control is silently inert (Dynamo.h:40) — a worse failure than a missing control because the GUI advertises tuning that never happens; and `Presets<…,0>` half-supports the empty set (its `next()/prev()` guard against `Size==0` while its accessors index a zero-length array, which is UB), so the interface's own contract is ambiguous.

### Performance & Resource Discipline — **A**
A standout, and clean this pass. Compile-time resolution dispatch, `FunctionRef` zero-allocation type erasure, split-trig LUTs, fast-reject heuristics, the fused three-species graph-Laplacian walk, branchless saturating color math, `fast_cbrt`/`fast_sinf` deployed exactly on the OKLab hot path and nowhere else, a single instanced draw call with GPU-side normalize, and worker parallelism measured as `max()` not `sum()`. Memory is rigorous to the byte. No new waste was found; the two redundant-compute nits this pass surfaced (`std::round` over an int RNG, an `snprintf`-then-log) are on cold paths and counted under documentation/consistency, not here.

### Documentation & Comments — **A**
Still the project's strongest instinct — comments encode *why*, not *what*, and the 1,900-line README functioned again as an accurate contract that fifteen reviewers used to reconstruct intent. Stepped down from the prior pass's A+ because this audit found comments that are not merely thin but *affirmatively wrong*: Dynamo's `TRAIL_CAPACITY` comment states "the runtime Trail Len slider sets the active length within this bound," describing wiring that does not exist (Dynamo.h:204-205), and the `test_death.h` header enumerates the death-suite seams but omits the Plot case it actually registers (test_death.h:6). A false comment is worse than a missing one because it directs the next reader *past* the very bug it papers over — which is precisely the standard this project holds itself to.

### Testing & Verification — **A−**
The verified fail-fast death harness (re-exec, exact illegal-instruction status), the byte-identical determinism harness, the auto-generated smoke pass that constructs and renders every effect with asserts on, and the host-testable wire-format/segment-tiling layers remain a genuinely rigorous net — fronted by a pre-commit hook, presubmit CI, and a gated deploy. But this pass is itself evidence of the net's blind spots: nothing caught the BZ species-C starvation (a determinism harness sees a *consistent* render, not a *correct* one — the dead arm is invisible to a byte-exact oracle), nothing caught the inert Dynamo slider (no test drives a live param and asserts the trail responds), and the death suite that *does* guard the degenerate-mesh class doesn't exercise `classify_faces` on a sub-triangular face. The seams are exactly where an automated oracle has nothing to compare against.

### Consistency with Stated Philosophies — **A−**
The five philosophies (16-bit linear color; compile-time resolution; fail-fast cold-only `HS_CHECK`; arena allocation; double-buffer-by-design) are still applied with rare uniformity and enforced at build/startup. The deductions are, again, the precise spots where the code deviates from its *own* rule: `operator*` does not mirror the NaN-safe `hs::clamp` its sibling `blend_alpha` adopted (and whose comment claims the mirror already exists); `to_short` clamps in the int domain where the blessed `float_to_pixel16` clamps in float; `classify_faces` half-applies a guard where the rest of `mesh.h` traps; and `wasm.cpp` passes a runtime buffer as a printf format two lines below a comment forbidding exactly that. Each is a local lapse in an otherwise-enforced discipline — fixable nits, not systemic drift — but there are more of them this pass than last.

## 4. Prioritized Fix List

*Each item was independently verified against the source by a separate validator and re-checked here. Severity: P0 = mis-paints real hardware or a latent semantic trap that fires on shipped input; P1 = robustness / verification / resource risk; P2 = consistency & correctness in live code; P3 = doc drift & cleanup. Items are numbered sequentially across the whole list.*

**P0 — correctness on real hardware / latent semantic traps that fire on shipped input**

*None found in scope this pass. Every candidate that survived validation is either confined to a code path the shipped product does not reach (the mesh and flower items) or fails only on already-anomalous input (the NaN clamp). The prior audits' hardware/feedback P0s remain remediated.*

**P1 — robustness / verification / resource risks**

*None survived validation as P1. The candidates that might have ranked here on first read — the `classify_faces` trap and the `operator*` NaN cast — were re-graded P2 by the validators because the former is unreached by registered geometry and the latter triggers only on an already-pathological NaN input.*

**P2 — correctness & consistency in live code**

1. **BZ stochastic perturbation never touches species C (dead ternary arm).** `perturb_state()` picks a target with `int s = hs::rand_int(0, 2)`, which is half-open `[0, 2)` and so yields only `{0, 1}`; the following `(s == 0) ? nA : (s == 1) ? nB : nC` therefore never selects `nC`, and species C of an explicitly cyclic A→B→C→A competition receives zero anti-convergence noise — the opposite of the perturbation's stated purpose, plus an unreachable arm. *Fix:* `hs::rand_int(0, 3)`; optionally add a test asserting all three buffers can be perturbed. `effects/BZReactionDiffusion.h:141`.

2. **`classify_faces_by_topology` half-applies its `count >= 3` guard.** Per-face corner angles are computed only when `count >= 3`, but the topology-hash loop reads `angles[k]` for all `k ∈ [0, count)` unconditionally; a degenerate (1- or 2-vertex) face — which the WASM `classifyFaces()` path can receive because `fromData()` bounds face vertex counts above (`UINT8_MAX`) but not below — reads stale scratch under `NDEBUG` and yields a non-deterministic topology id instead of trapping. *Fix:* move the hash-combine loop inside the existing `if (count >= 3)` block, or add `HS_CHECK(count >= 3)` to trap degenerate input outright, matching `narrow_face_count` elsewhere in the file. `core/mesh.h:404-422`.

3. **`Plot::Flower` planar projection basis is centered on the wrong hemisphere.** For `radius < 1` the petal ring clusters near `−work_basis.v`, but `Flower::draw()` builds the planar projection basis on `+work_basis.v` — the opposite pole — so the azimuthal-equidistant projection that gives the flower straight tangent-plane edges is severely distorted (and below `radius ≈ 0.028` the per-segment antipode guard forces geodesic strategy entirely). `Scan::Flower`/`SDF::Flower` correctly center on `−basis.v`, confirming Plot's center is the inconsistent one; the antipode fallback prevents any NaN, which is why the degradation is silent. *Fix:* center the planar basis on `−work_basis.v` (mirroring `SDF::Flower`), with a one-line comment noting the projection center is the antipode of the work basis. `core/plot.h:1279`.

4. **Dynamo "Trail Len" slider is a silent no-op.** The slider registers `&params.trail_length`, but that value is consumed exactly once — at member-initializer time, passed into `Filter::World::Trails<W,…>`, which stores `lifetime` as a const-after-construction private member with no setter — so moving the slider after init has zero effect. Dynamo is the only effect exposing a Trail-Len control, so this is a lone divergence from the engine's otherwise-careful live-param wiring. *Fix:* add a `set_lifetime(int)` to `World::Trails` (same `>0 && <=255` check as the ctor) and call it each frame from a clamped `params.trail_length`; or drop the dead `registerParam` and member if live tuning is out of scope. `effects/Dynamo.h:40`, ctor at `effects/Dynamo.h:26-27`.

5. **`Pixel16::operator*` clamps NaN-unsafely before a `uint16_t` cast.** Each channel is clamped with `std::clamp(r*s, 0.0f, 65535.0f)` — which returns NaN unchanged — and then cast to `uint16_t`, an undefined float→int conversion on a NaN scale. This is the exact hazard deliberately fixed in `blend_alpha` (which switched to the NaN-mapping `hs::clamp` and is locked by `test_blend_alpha_clamps_before_cast`); `operator*` is the more central primitive (it backs `Color4::operator*=`, `blend_accumulate`, and every `Pixel * float`), so the codebase's NaN-safety guarantee is only partial. *Fix:* replace `std::clamp` with `hs::clamp` in the three channel expressions — bit-identical for valid inputs, defining only the NaN path. `core/color.h:85-87`.

**P3 — doc drift & cleanup**

6. ✅ **`to_short` casts float→int before clamping.** `std::clamp(static_cast<int>(std::round(t * 65535.0f)), 0, 65535)` performs the cast before the int-domain clamp, so a NaN/overflow input is UB before the clamp can fire; all three live callers feed pre-sanitized values today, so it is latent, but it diverges from the blessed `float_to_pixel16`, which clamps in float space first. *Fix:* `static_cast<uint16_t>(hs::clamp(t, 0.0f, 1.0f) * 65535.0f + 0.5f)`. `core/color.h:366-369`.

7. ✅ **Redundant `std::round()` over an already-integer RNG result.** `RandomTimer::reset()` computes `t + static_cast<int>(std::round(hs::rand_int(min, max)))`, but `hs::rand_int` already returns an `int`, so the round is a dead no-op that misleadingly implies a fractional result; it is also the only one of ~25 `rand_int` call sites that rounds. *Fix:* `next = t + hs::rand_int(min, max);`. `core/animation.h:505`.

8. **`Presets<…, 0>` accessors are UB while `next()/prev()` guard the empty case.** `next()` and `prev()` both `if constexpr (Size == 0) return;`, but `get()`, `apply()`, and `prev_get()` index a zero-length `std::array` unconditionally; no call site instantiates `Size == 0` today, so it is a latent template trap with an inconsistent contract. *Fix:* add a class-level `static_assert(Size > 0, …)` (the empty case is meaningless for a preset selector) and drop the now-unneeded `Size == 0` branches. `core/presets.h:19`.

9. **Dynamo comment claims a runtime trail-length control that does not exist.** The `TRAIL_CAPACITY` comment states "the runtime Trail Len slider sets the active length within this bound," describing wiring the code does not implement (see item 4) — an affirmatively false comment that will direct a maintainer past the bug. *Fix:* once item 4 lands the comment becomes true; until then, state that `TRAIL_CAPACITY` is the fixed storage bound and the active length is captured once at construction. `effects/Dynamo.h:204-205`.

10. **WASM init-HWM log passes a runtime buffer as the printf format string.** `setEffect()` does `snprintf` into `hwm_buf` then `hs::log(hwm_buf)`, passing a runtime buffer as the format argument of a `format(printf,1,2)` function — directly against the convention the same file documents 30 lines above and applies correctly elsewhere. Harmless today (the literal is `%`-free) but a latent format footgun and a redundant two-step. *Fix:* `hs::log("WASM: init stack HWM = %u bytes", (unsigned)stack_high_water_mark());`, dropping `hwm_buf`. `targets/wasm/wasm.cpp:222-225`.

11. **Death-test header seam enumeration omits the Plot case.** The `test_death.h` file-level comment lists the death-suite seams (memory/arena, math, mesh-topology, registry, container, spatial, animation, canvas, scan) but omits `plot`, even though `case_plot_mesh_vertex_over_capacity` is defined and registered in `all_cases()` — so the canonical coverage index understates what the suite actually guards. *Fix:* add `plot` to the enumerated seam list. `tests/test_death.h:6`.

### 4a. Verified and dismissed this pass

*Recorded so they are not re-raised in a future audit.*

- **`ease_in_out_bicubic` misnamed (claimed P3)** — *rejected: below threshold.* The function implements a standard 1D `easeInOutCubic`, so "bicubic" (conventionally 2D) is genuinely loose terminology, but the adjacent comment correctly says "Cubic Interpolation," the signature is scalar `float(float)` so no caller could ever misuse it as a 2D interpolator, and the name is load-bearing across the README and two test files. Renaming is non-trivial churn for zero functional benefit.
- **`vectorToLogPolar` south-pole `−inf` asymmetry (claimed P3)** — *rejected: dead code.* The north pole gets a finite sentinel while the south pole computes `rho = −inf`, but the round-trip still works (`expf(−inf) = 0`) and a tree-wide grep finds zero production call sites — only the definition and its own test. Documenting a function nothing calls is not worth a developer's time; matches the prior `closest_point_on_mesh_graph` dead-code disposition.
- **HD107S color-correction truncation (claimed; reviewer-graded the component A− on it)** — *rejected: by-design.* The `>> 8` correction multiplies do truncate where `color.h` rounds, but the hardware *wire* path deliberately mirrors FastLED's fixed-point `scale8` semantics (documented at `platform.h:415-417`) so the simulator predicts the device; adding a rounding bias would *diverge* from FastLED, not converge. The "breaks sim/device parity" premise is also false — the sim's `setCorrection`/`setTemperature` mocks are no-ops, so the device already differs by the entire correction factor, dwarfing a sub-LSB rounding bias. A variant of the already-dismissed truncate-vs-round class, applied to a path that is correctly truncating.
- **BZ/GS seed-center RNG excludes the last lattice node (claimed P3)** — *rejected: sub-threshold.* `rand_int(0, RD_N-1)` never selects node 7679 as a seed *center*, but that is 0.013% of nodes on a stochastic lattice that re-seeds every reset, and node 7679 is still reachable as a *neighbor* of seeded centers. No visible effect; the genuine BZ defect here is item 1 (species C), a different line.
- **Daydream dead exported label helpers (`coordsLabel`/`prettify`, claimed P3)** — *rejected: dead code.* ~34 lines of exported helpers with zero importers; `getLabels()` (their only would-be consumer) is defined nowhere. The one real hazard (raw `<br>` markup) is already closed by the `textContent` sink, so the issue is purely latent in unreachable code — matches the established dead-code disposition.

*(Carried forward from prior audits and still not defects — do not re-raise: the `normalize()`/`normalized_or()` "epsilon inversion"; the WASM `getVertices()` "use-after-free"; the `BZReactionDiffusion::blend_species` "formula"; the `FunctionRef::operator()` hot-path `assert`; the StereoShaderBase dedup; the live-param-sync factoring; the reaction-graph `PROGMEM` placement; the Liquid2D `Driver` wrap; the double-buffer "tearing" non-issue; the `closest_point_on_mesh_graph` dead-code orbit guard; the IslamicStars truncate-vs-round; and the `SDF::Face` constructor init-order "opacity.")*

## 5. Overall Impression

Three prior audits converged on a single observation: that independent readers handed disjoint slices kept returning the same account of one engineer's values. That remains true, and this pass re-confirms it from fifteen fresh vantage points. The more interesting result of going a layer deeper is what it reveals about *which* defects a verification regime this strong still lets through — because the eleven that survived are not randomly distributed. They cluster, almost perfectly, in the one place the project's automated oracles are structurally blind: **outputs that no machine can tell are wrong.**

Consider the failure modes. The determinism harness proves a render is *byte-identical on replay* — but a Belousov–Zhabotinsky field with one of three species quietly starved of perturbation is still perfectly deterministic; it just animates with a subtle asymmetry no diff can see. The death harness proves the cold-path traps *fire* — but a flower whose planar projection silently falls back to a geodesic polygon never trips a trap; it just draws a slightly-wrong-but-plausible shape. The smoke pass proves every effect *constructs and renders without asserting* — but a "Trail Len" slider that does nothing renders fine; its breakage only exists in the gap between what the UI promises and what a human, dragging it, expects. Every one of these is a place where correctness is defined by *intent and human perception*, not by an invariant a test can encode. The engine's working core — math, memory, the pipeline, the drivers — returned no findings precisely because there correctness *is* expressible as an invariant, and the author has encoded it.

So the honest read of this pass is not that the tree got worse; it is that the audit got deeper and found the residue exactly where the methodology predicts it must live. A solo author with a world-class automated net will be cleanest where the net can grade him and weakest where only a second human eye — or a user's hand on a slider — could have caught the slip. That is not a indictment of the discipline; it is the discipline's own signature, read off its shadow. None of the eleven is rot. All of them are the specific debt of a one-person team whose tests can check arithmetic and lifetimes but cannot check *taste, intent, or interaction*. The fix list is short and mechanical because the values underneath it are intact; the items survived only because they sit in the blind spot no harness can illuminate.

## 6. Technical & Artistic Merit — The Achievement

The most useful lens on this artifact is that **its constraints are the co-author.** A microsecond ISR budget, 335 KB of RAM, no attached debugger, a strip of LEDs smeared into a sphere by 480 RPM of rotation — these are not obstacles the engine works around; they are the forcing function that produced its best ideas. Compile-time `<W,H>` specialization exists because a Cortex-M7 cannot afford runtime generality. The arena allocator exists because the Teensy heap fragments. The lock-free triple-index double buffer exists because the ISR cannot wait on the renderer. The fail-fast `HS_CHECK` doctrine exists because a corrupted frame on a spinning sphere is a silent, already-gone failure with no console to read. Strip the constraints away and most of what makes this codebase elegant would have no reason to exist. It is a sustained demonstration that *scarcity, taken seriously, is a better architect than abundance* — and the engineering reads that way: every clever mechanism is downstream of a hard physical limit the author refused to paper over.

The artistic layer earns the same respect because it is mathematically *load-bearing*, not decorative. These effects implement the actual Hopf fibration of S³ with a 4D tumble, the associated-Legendre recurrence for real spherical harmonics, Belousov–Zhabotinsky and Gray–Scott reaction–diffusion on a precomputed 7,680-node Fibonacci K-NN graph, authentic Hankin-method Islamic star patterns generated from Conway-operated Archimedean solids, twisted-torus volumetrics with an analytic Lipschitz bound for safe sphere-tracing, and Möbius-warped grids with pole-stabilizing counter-rotation. The tell is the singularity handling: someone who merely *copied* these formulas would not know to guard the gnomonic antipode, the conformal-mapping poles, the `acos` domain edges — and the recurring theme across all four audits is that the author guards them by reflex, getting right the part that is genuinely hard. (That this pass found the *one* flower whose projection center is on the wrong pole only sharpens the point: it stands out precisely because everything around it is correct.) The art is written by someone who understands the geometry, and the geometry is what the device exists to paint. The achievement is not any single effect; it is that the mathematics, the firmware, and the verification are the same body of code, held at bit parity across a microcontroller and a browser.

## 7. Comparative Appraisal — Independent Assessment

*The prior audit asked, of each peer class, "what does each side structurally lack that the other has?" This assessment asks a different question — the one this fourth pass kept surfacing: **how do you know any of it is true?** Reproducibility and falsifiability are the currency, because that is the axis on which this artifact is unusually, almost anomalously, far ahead — and the one on which most of its peers are quietly weak.*

**The maker / addressable-LED world (FastLED, WLED, one-off POV rigs).** A typical effect in that ecosystem is *true by demonstration*: it looks right on the bench, so it ships. There is rarely a way to re-derive whether last week's build painted the same frames, because the render is a function of wall-clock, an unseeded RNG, and a specific board. Holosphere's effects are *true by replay* — a mock clock, a pinned `mt19937(1337)`, and a byte-exact determinism harness mean any claimed frame can be regenerated and diffed years later. That ecosystem has breadth, community, and thousands of board configs Holosphere will never match; what it structurally cannot offer is the ability to *prove a render is the one you meant*. On the engineering-verification axis the gap is categorical, and it is the axis this fourth pass leaned on hardest.

**Professional creative-technology tooling (Pixelblaze; MadMapper / TouchDesigner; architectural-LED controllers).** These products are *true by team and calendar*: a QA org, soak tests, fuzzing, and headcount stand behind every claim. Holosphere reproduces the *practices* a serious graphics team would recognize — host-testable protocol layers, a death harness that asserts the exact trap status, build-provenance hashing, a reflection-driven GUI — at a fraction of the headcount, inside one narrow scope. Where it visibly falls short of a hardened product is exactly where this pass found its live defects: in the human-facing and interaction surfaces (an inert slider, a false comment) that a QA pass driving the UI would have caught in an afternoon, and that no determinism harness ever will. The core is product-grade; the *interaction seams* are single-authored, and they show it.

**Academic / research code.** Research artifacts are often *true on paper and unrebuildable in practice* — a brilliant result behind a prototype no one else can run. Holosphere inverts the usual research failure mode: it is **integrative, not novel** (OKLab, SDF CSG, sphere tracing, the Hopf fibration, Conway operators are all established art), but it is *survivable and reproducible* in a way the median research prototype is not. Its standout virtue as a research artifact is methodological — the determinism harness and the bit-faithful twin make every claimed frame replayable — not algorithmic. It would not pass a novelty bar at a graphics venue; it would embarrass most of those venues on reproducibility.

**The state of the art.** It does not move the rendering or color-science frontier and does not try to. Where it plausibly *is* at a frontier is the specific conjunction it occupies and proves: a perceptually-correct, compile-time-specialized, formally-fail-fast renderer running bit-identically across a four-microcontroller POV sphere and a WebAssembly browser twin, every invariant death-tested and every frame replayable. The frontier it touches is one of *integration and verifiability*, not of any single primitive — and it would be genuinely surprising to find a more rigorously reproducible open instance of that exact intersection.

**Standing.** As solo work this is exceptional, and four audit rounds — the last two spent mostly *retiring* candidate findings — bound the read tightly. The ceiling between this **A−** and an A is not set by anything present in the tree; it is set by the one thing a solo author with a great automated net cannot supply himself: **a second pair of eyes on the surfaces the net cannot grade.** This pass is a small proof of that thesis — it found a starved species, a wrong-pole flower, and a dead slider not because the author is careless but because those are exactly the defects that survive a regime built to check arithmetic and lifetimes. Land the five P2 items and the distance closes; the work underneath them was never the problem.
