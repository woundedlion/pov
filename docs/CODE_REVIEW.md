# Holosphere ‚Äî Code Quality Review

*Fifth independent multi-agent audit, 2026-06-10. The codebase was partitioned into eighteen component groups spanning the full C++20 engine (`core/`, `effects/`, `hardware/`, the WASM bridge, the C++ test suite, the build/CI plumbing) and the sibling `daydream/` JavaScript simulator (renderer, GUI/recorder, segmented-POV worker pool, geometry tools). One reviewer read every file in its group against the project's intent; **89 candidate findings** were then each handed to a separate adversarial validator that re-opened the source and weighed it for **truth, relevance, and whether a fix is worth a developer's time**. **57 survived; 32 were dismissed.** A final synthesis pass aggregated the per-component grades and the surviving findings into the report below. Out of scope by request: `core/effects_legacy.h`, `targets/Holosphere/Holosphere.ino`, and `core/rotate.h`; `core/FastNoiseLite.h` is treated as vendored, and the generated tables (`core/color_luts.h`, `core/reaction_graph.cpp`) were spot-checked rather than read line by line. Letter grades (A+ ‚Ä¶ F) are measured against a senior / graphics-middleware professional bar ‚Äî not "does it run."*

*This pass post-dates a fourth audit recorded earlier the same day, and its picture has been **verified to have changed underneath it**: every open P2/P3 recommendation that audit raised has since landed in the tree ‚Äî BZ's perturbation now draws `rand_int(0, 3)` (species C reachable), `Pixel16::operator*` now clamps with the NaN-safe `hs::clamp`, `World::Trails` gained a `set_lifetime` setter, `to_short` now clamps in float space, and `Presets` carries `static_assert(Size > 0)` ‚Äî and the ¬ß4a `ease_in_out_cubic` rename, the `vectorToLogPolar` south-pole sentinel, and the daydream label-helper HTML fix are committed. Two of this pass's findings (#16, #23) are in fact **direct consequences** of those very fixes. What is genuinely new here is breadth: this audit ran the daydream JS test suite (finding one red test the prior pass never executed), reached the WASM/JS bridge and the firmware frame-sync startup, and so located surviving debt at the **integration seams** the narrower passes did not exercise. The ¬ß¬ß "Reconciliation with the Prior Audit", "Overall Impression", "Technical & Artistic Merit", and "Comparative Appraisal" below are formed independently and deliberately do **not** reuse the prior audits' framing or conclusions.*

## Executive Summary

Holosphere is a genuinely high-caliber piece of systems engineering: a single C++20, compile-time-`<W,H>`-templated rendering engine that drives both a 480 RPM persistence-of-vision LED sphere (Teensy 4.x firmware) and a browser WASM simulator (`daydream`, Three.js). Judged at a senior/professional bar, the codebase is in the top tier for correctness discipline, numerical care, documentation density, and test rigor. The design is opinionated and coherent ‚Äî arenas because the Teensy heap fragments, always-on `HS_CHECK` traps because there is no on-device debugger, exact FastLED mocks so the simulator predicts the device, and zero-runtime-generality compile-time specialization throughout.

The engine core (math, color, SDF/scan, plot/filter, animation, mesh, memory, hardware drivers) is excellent and largely defect-free; the confirmed bugs there are latent and low-impact. The real concrete defects cluster at the **seams** ‚Äî the WASM/JS bridge, the segmented-POV worker pipeline, the firmware frame-sync startup, and effect parameter extremes ‚Äî which is exactly where a system this internally-consistent tends to leak. The most actionable items: a WASM `fromData` missing-`UINT16_MAX`-bound that silently truncates indices and then aborts the whole module (high), a **currently-failing** `daydream` worker unit test on the most data-flow-critical path (high), a reachable timeline-event overflow in ShapeShifter that silently drops visuals (medium), a narrow-but-latching frame-sync parity-seed hazard on downstream boards (medium), and a screenshot-tooling effect list that has already drifted from the registry (medium).

The web layer is polished and careful for browser code (detached-buffer guards, aliased zero-copy display buffer with asserted invariants, thorough teardown), but its JS test coverage is thin outside `state.js` and the codec picker, and `palettes.html` is a maintainability outlier.

**Overall grade: A-**

### Correctness ‚Äî A-
The engine math is correct and verified by behavior-focused, oracle-driven tests: stereo/gnomonic round-trips, quaternion identities, Euler-characteristic checks across four Platonic seeds, brute-force KD-tree cross-checks, and sorted-interval CSG regressions. Pole/antipode/singularity handling is systematic. The deductions are concrete and concentrated at boundaries: the WASM `fromData` 16-bit index gap (high), a failing worker compositor test (high), ShapeShifter's `66 > 64` timeline overflow (medium), the frame-sync parity-seed hazard (medium), plus latent low-impact bugs (AngularRepeat non-Y vertical bounds, Flower pole wraparound, `Color4::lerp` alpha/color clamp asymmetry, `BakedPalette::get` NaN UB, the Lissajous export argument-order mismatch).

### Architecture & Design ‚Äî A
Compile-time `<W,H>` specialization, single-arena partitioning with explicit `Arena&` threading, the variadic `Pipeline<W,H,Filters...>` with `if-constexpr` domain transitions, the half-edge mesh core reduced to shared orbit/edge-walk primitives, and the X-macro single-source-of-truth effect roster form a tight, intentional whole. `daydream` mirrors it with single-writer URLSync, lazy-getter worker injection, and a one-frame segmented pipeline matching the multi-Teensy hardware model. Minor smells only: Liquid2D/Flyby skeleton duplication, the dead `setSegment` protocol path, and a slightly overloaded driver class.

### Interface Expressiveness & API Design ‚Äî A-
Strong intent-revealing design (named factories, `normalized_or` split, deleted-rvalue borrow guards, typed owned-vs-borrowed spans, boolean-returning WASM setters). Held below A by under-specified or fragile surfaces (`inv_gnomonic` sign overload, per-shape `DistanceResult` field semantics) and a recurring dead-API theme across both repos.

### Code Readability & Maintainability ‚Äî A-
Comments explain the *why* to an unusually high standard. The recurring drag is duplication and dead stores (`static inline` easing, dead pre-C++20 `lerp`, `to_short`/`float_to_pixel16` twins, `Mutation::from`, triplicated mesh test fixtures) and one genuine outlier: `palettes.html`, a 1500-line inline monolith littered with `NEW:`/`MODIFIED:`/`FIXED:` change-log comments that violate the project's own no-rename-history convention.

### Performance & Efficiency ‚Äî A
Disciplined hot/cold separation, fixed-point and LUT-backed hot paths with measured error budgets, zero per-pixel allocation, single-DMA-transfer hardware path, zero-copy WASM/worker transfers. Minor: per-ring `make_basis` recompute, per-point Comets rasterization, and unbounded float phase accumulation that degrades trig precision on multi-hour runs.

### Memory Safety & Resource Management ‚Äî A-
Wrap-safe arena math, two-axis use-after-free tracking, multiply-site overflow traps, RAII evacuators, trapping narrows, and diligent GPU/WASM disposal. The one concrete gap is the `fromData` silent `uint16` narrowing; latent sharp edges are `ArenaSpan`-from-temporary, `clone_from` null-source memcpy, and a resize-time material/uniform aliasing fragility in `driver.js`.

### Error Handling & Robustness ‚Äî A-
The fail-fast doctrine is applied with precision and the untrusted JS/WASM boundary correctly rejects-rather-than-traps ‚Äî *except* the one `fromData` hole. Other gaps: ShapeShifter relying on Timeline's silent drop at a reachable cap, `Trails` negative re-emitted age after `set_lifetime` shrinks, and `solids.html` rendering a partial mesh after a caught op error.

### Testing & Verification ‚Äî A-
The C++ host suite alone would earn an A: analytic/oracle/invariant assertions, signature-pinned regressions, a death harness asserting the *exact* trap signal, and roster-generated coverage that cannot drift. The aggregate drops to A- for systemic gaps ‚Äî a failing JS worker test, near-zero coverage of `driver.js`/compositor/geometry math, no runtime exercise of the shipped WASM module, untested `fromData` rejection paths, and several untested heavier animation types.

### Documentation ‚Äî A
Load-bearing, accurate inline documentation across both repos, with cross-referenced contracts (WASM memory-view detachment mirrored in JS, provenance threat model, fast-math flag rationale). Isolated gaps: the H_OFFSET device/host divergence documented only in a test comment, `DistanceResult` per-shape semantics, a misplaced `read_id` comment, and the `palettes.html` comment noise.

### Portability & Build System ‚Äî A-
Clean three-way platform split, correct and rare `-ffast-math`/`-fno-finite-math-only` handling, an exemplary WASM provenance trio gated on the recorded engine SHA, pinned toolchain/actions, and robust Windows-Clang detection. Deductions for the H_OFFSET parity gap, the ODR-fragile header-scope `instance_`, host divide-by-zero edges, and the silent `build_release.bat` install no-op.

### Architectural Elegance ‚Äî A
Split-table LUT reconstruction, sentinel-unified projection algebra, shared SDF interval emitters, per-face Lipschitz-bounded LUTs, compile-time `StaticPalette` folds, `opaque()`-defeated death tests, self-referential determinism checks, and the self-locating build-time importmap are all tasteful. A few seams (global-constant sprawl, duplicated skeletons, dead protocol members) keep it just shy of flawless.

## Prioritized Fix List

### Critical

*(none ‚Äî no defect rises to data-loss, security, or guaranteed-crash-in-normal-use severity; the firmware's fail-fast traps are intentional)*

### High

1. ‚úÖ **WASM `fromData` omits `UINT16_MAX` vertex/index bound** ‚Äî `targets/wasm/wasm.cpp:468-581` ‚Äî a >65535-vertex/index editor mesh fits the 8 MB tooling arena, silently narrows `int`‚Üí`uint16_t` at the `m.faces.push_back(idx)` (line 572), corrupts topology, then aborts the whole module on the next operator via `build_from_flat`'s `HS_CHECK` ‚Äî defeating the file's own reject-don't-trap intent. ‚Üí After computing `num_verts`/`num_face_verts`, add `if (num_verts > UINT16_MAX || num_face_verts > UINT16_MAX) { hs::log(...); return nullptr; }` alongside the other validations.
2. ‚úÖ **Failing `daydream` worker unit test (ArrayBuffer vs Uint16Array)** ‚Äî `daydream/tests/segment_controller.test.js:86` ‚Äî `deliverFrame` sends `pixels: px.buffer` but the protocol/worker send a `Uint16Array`; after commit `c551007` dropped the controller copy, `composite()` indexes the ArrayBuffer, gets `undefined`‚Üí0, and the `tick()`‚Üícomposite assertion FAILS on `npm test`, red-flagging CI and removing real coverage of the most data-flow-critical path. ‚Üí Change the helper to `pixels: px` (drop `.buffer`) so the mock mirrors the protocol typedef, then re-run `npm test`.
3. ‚úÖ **AngularRepeat vertical bounds wrong for non-Y axis** ‚Äî `core/sdf.h:977-979` ‚Äî `get_vertical_bounds` forwards the un-repeated child's latitude band, so folded copies on any non-Y axis (the README-advertised arbitrary-axis path) fall outside the row-clip and are silently never rasterized. ‚Üí Detect `fabsf(axis.y) < 1 - TOLERANCE` and return `{0, H-1}` (conservative full-canvas) for non-Y axes; keep tight forwarding only for the Y specialization.

### Medium

4. **ShapeShifter overflows the 64-slot global timeline at Count=16** ‚Äî `effects/ShapeShifter.h:74` ‚Äî `2 + 4*num_shapes = 66 > 64` events; `Timeline::add()` silently log-and-drops the last two, leaving a ring with no rotation/Sprite (a missing/static shape) instead of failing loudly. ‚Üí Lower the registered `Count` max to 15 (or `HS_CHECK(2 + 4*num_shapes <= MAX_EVENTS)` in `rebuild()`).
5. **Frame-sync parity can seed to the wrong boundary at effect start** ‚Äî `hardware/pov_segmented.h:444` ‚Äî a downstream board re-zeros `x_` against an arbitrary sphere phase, so proximity seeding can name the opposite boundary and latch a permanent half-revolution arm-swap for the whole effect with no self-correction. ‚Üí Seed parity from an unambiguous signal (distinct pulse width / second sync line, or defer seeding until the shared clock has driven `x_` past one observed interval), or gate the seed on provable `< w/4` drift.
6. ‚úÖ **Screenshot effect list is a hardcoded registry duplicate that has drifted** ‚Äî `scripts/capture_screenshots.mjs:13` ‚Äî the 26-name `EFFECTS` array omits `DistortedRing` and `ShapeShifter` (both `REGISTER_EFFECT`), so they never get a gallery screenshot and any new effect is silently skipped. ‚Üí Derive the list from the running simulator's registry (the GUI already enumerates it), or add a CI assertion that the list equals the registry. **Fixed:** the script now parses the `HS_EFFECT_LIST` X-macro in `core/effects.h` (the same single-source roster the WASM startup check and native smoke suite derive from), so the gallery can never drift from the registered set.
7. ‚úÖ **No test ever runs the shipped WASM module at runtime** ‚Äî `.github/workflows/ci.yml:94` ‚Äî the wasm job is compile-only and the deploy gate runs the *native* suite, so a SIMD-codegen, embind-mismatch, 8 KB-stack-overflow, or `ALLOW_MEMORY_GROWTH` regression would pass CI and deploy. ‚Üí Add a headless Node smoke step that imports the MODULARIZE/ES6 module, steps a few frames across effects, and asserts `getArenaMetrics()` stack high-water-mark < `STACK_SIZE`. **Fixed:** added `scripts/wasm_smoke.mjs` (a dependency-free Node ES6 importer of the built `holosphere_wasm.js`) and a CI step that runs it right after the wasm build. The script enumerates effects from the running registry (`getEffectSizes()` ‚Äî same drift-proof source the screenshot gallery uses), steps three frames per effect at both `96x20` and `288x144`, asserts each `getPixels()` view has the full active-resolution length (catching an `ALLOW_MEMORY_GROWTH` detachment), and asserts every `getArenaMetrics()` region ‚Äî the four arenas **and** the stack ‚Äî keeps its high-water mark within capacity (so the tight 8 KB stack overrun the finding names fails the build). The capacity is read from the module itself rather than hardcoding `STACK_SIZE`, and the worst-case stack margin is logged. A `wasm-smoke` npm script invokes it locally; verified against the installed module (peak stack 616/8192 bytes, all effects green).
8. ‚úÖ **`fromData` input-validation paths have no host test** ‚Äî `targets/wasm/wasm.cpp:468-542` ‚Äî the most input-fragile code in the bridge (five reject branches incl. the missing `UINT16_MAX` case) lives behind embind and is untested, which is how the index gap survived. ‚Üí Extract the pure build-and-validate logic into a header sibling of `param_marshal.h` and cover each reject branch like the existing param-marshal tests. **Fixed:** the validate-and-build logic now lives in `targets/wasm/mesh_marshal.h` (`build_mesh_from_flat`, returning a `MeshBuildStatus`), with `fromData` reduced to JS-array conversion + a single status check. `tests/test_mesh_marshal.h` covers all five reject branches ‚Äî non-multiple-of-3 verts, out-of-range face index, >`UINT8_MAX` face size (with the 255/256 boundary), the `UINT16_MAX` vertex/half-edge index bound (finding #1's branch, relocated into `build_mesh_from_flat` so it is host-tested rather than living inline behind embind), and arena overflow ‚Äî plus the "arena untouched on reject" invariant and the face-counting edge cases (empty/leading/consecutive delimiters, no trailing delimiter).

## Overall Impression

*Independent assessment ‚Äî not based on the prior review.*

The strongest single impression this codebase leaves is **coherence under a constraint set most projects would treat as a reason to lower their standards.** A 600 MHz Cortex-M7, 335 KB of usable RAM, a microsecond-scale ISR budget, no attached debugger, and a strip of LEDs that only becomes an image because it is spun at 480 RPM ‚Äî these are conditions under which "good enough to light up" is the normal ceiling. Holosphere instead treats each limit as a design premise and answers it with a specific, defensible mechanism: compile-time `<W, H>` specialization so the device pays nothing for the simulator's generality; a single partitioned arena because the heap fragments under mesh subdivision; a lock-free triple-index double buffer because the ISR cannot block on the renderer; and an always-on `HS_CHECK` trap doctrine because a silently corrupted frame on a spinning sphere is the worst possible failure ‚Äî gone before it can be seen, with no console to read it off. The result is not a pile of optimizations; it is a system whose parts all point the same direction.

What elevates it past "well-engineered firmware" is that the same C++ runs **bit-identically** in a browser. The WASM twin is not an approximation of the device ‚Äî it shares the effect classes, the arena, the per-frame `Pixel16[]` buffer, the deterministic `mt19937(1337)` seed, and even mocks FastLED's fixed-point color math so the simulator *predicts* what the hardware will paint. That single decision is what makes the whole verification story possible: a determinism harness can diff frames across years, a death harness can assert the exact illegal-instruction status of every cold-path trap, and a contributor can debug a 288√ó144 effect in Chrome and trust it on four Teensys. The engineering and the art are the same body of code held at parity across two radically different machines, and that is rare at any team size, let alone for solo work.

The audit's honest counterweight is that the polish is not uniform. The core ‚Äî math, color, memory, the rasterizers, the pipeline, the drivers ‚Äî is where correctness is expressible as an invariant, and there the author has encoded it to a standard that returns almost no findings. The debt clusters precisely where correctness is defined by *intent, perception, or a cross-module contract* that no local invariant captures: a worker-pool composite test that is red right now, a bridge index that narrows silently on an over-large editor mesh, a parameter extreme that overflows a fixed timeline, a UI surface or two. None of it mis-paints shipped hardware; all of it is cheap to fix; and its location is not random ‚Äî it is the shadow cast by a verification regime that is world-class at grading arithmetic and lifetimes and necessarily blind to taste, interaction, and the seams between subsystems. The fix list is short and mechanical because the values underneath the code are intact.

## Technical & Artistic Merit ‚Äî The Achievement

The technical achievement is an **integration feat executed at an unusually high craft bar**, not a new algorithm. Every primitive ‚Äî OKLCH perceptual color, signed-distance CSG, sphere-traced volumetrics with a Lipschitz bound, the Hopf fibration of S¬≥, associated-Legendre spherical harmonics, Belousov‚ÄìZhabotinsky and Gray‚ÄìScott reaction‚Äìdiffusion on a precomputed Fibonacci K-NN graph, Hankin-method Islamic star patterns over Conway-operated Archimedean solids ‚Äî is established art. The merit is in fusing them into one zero-allocation, fail-fast, compile-time-specialized engine that survives on a microcontroller and reproduces in a browser, and in getting the *hard* parts right by reflex. The tell, recurring across the whole tree, is the singularity handling: the gnomonic antipode, the conformal-mapping poles, the `acos` domain edges, the half-open RNG conventions, the NaN-before-cast hazards. These are exactly the places a copyist's port breaks, and they are guarded here as a matter of habit. Someone who merely transcribed these formulas would not know to defend them; this author defends them before being asked.

The artistic layer earns equal respect because it is **mathematically load-bearing rather than decorative.** The effects are not noise functions dressed up as physics ‚Äî they are the physics, rendered honestly: a real 4D tumble of the fibration, an actual three-species cyclic competition, genuine star-polygon intersections swept by the Hankin angle. That fidelity is also why the art is *credible* on the verification axis the rest of the engine prizes: because the math is real, a frame can be re-derived, and because it can be re-derived, it can be trusted. The achievement is not any one effect on the sphere; it is that the geometry the device exists to paint, the firmware that paints it, and the harness that proves it painted correctly are a single, internally consistent artifact.

## Comparative Appraisal ‚Äî Independent Assessment

*The axis that most cleanly separates this artifact from its peers is **reproducibility and verifiability** ‚Äî the ability to prove that a given frame is the one that was meant ‚Äî so that is the axis these comparisons are drawn on.*

**The maker / addressable-LED ecosystem (FastLED, WLED, one-off POV builds).** This is the natural peer group, and Holosphere is several tiers above its median on engineering rigor while conceding breadth and community. A typical effect there is *true by demonstration* ‚Äî it looks right on the bench, against an unseeded RNG and wall-clock time, so it ships, and there is usually no way to re-derive whether last week's build painted the same frames. Holosphere's effects are *true by replay*: pinned seed, mock clock, byte-exact determinism harness. What that ecosystem offers and Holosphere does not is thousands of supported board/LED configurations and a large community; what it structurally cannot offer is a proof that a render is correct, and on that axis the gap is categorical.

**Professional creative-technology tooling (Pixelblaze, MadMapper, TouchDesigner, architectural-LED controllers).** These are *true by team and calendar* ‚Äî QA orgs, soak tests, fuzzing, and headcount stand behind each release. Holosphere reproduces the *practices* a serious graphics team would recognize ‚Äî host-testable wire/protocol layers, a death harness asserting exact trap status, build-provenance hashing with a gated deploy, a reflection-driven GUI ‚Äî at a fraction of the headcount and within one narrow scope. Where it falls short of a hardened commercial product is exactly where this audit found its live debt: the human-facing and cross-module *interaction* surfaces (a red worker test, an under-validated bridge input) that a QA pass driving the system end to end would surface in an afternoon. The core is product-grade; the integration seams are single-authored and show it.

**Academic / research code.** Research artifacts are frequently *true on paper and unrebuildable in practice*. Holosphere inverts that failure mode. It is **integrative, not novel** ‚Äî it would not clear a novelty bar at a graphics venue ‚Äî but it is *survivable and reproducible* in a way the median research prototype is not, and its determinism harness plus bit-faithful twin would embarrass many published artifacts on reproducibility. Its standout virtue as research-adjacent work is methodological, not algorithmic.

**State of the art.** It does not advance the rendering or color-science frontier and makes no claim to. The frontier it plausibly occupies is a *conjunction*: a perceptually-correct, compile-time-specialized, formally-fail-fast renderer running bit-identically across a four-microcontroller POV sphere and a WebAssembly browser twin, with every cold-path invariant death-tested and every frame replayable. The frontier here is one of integration and verifiability, not of any single primitive ‚Äî and a more rigorously reproducible open instance of that exact intersection would be genuinely hard to find.

**Standing.** As solo work this is exceptional, and the convergence of five audit passes ‚Äî the last several spent mostly *retiring* candidates and confirming remediations ‚Äî bounds the read tightly. The distance between this **A‚àí** and an A is not set by anything wrong in the engine's core; it is set by the integration seams and the surfaces an automated oracle cannot grade ‚Äî the one thing a solo author with a world-class verification net cannot supply himself: a second pair of eyes on the parts the net does not reach. Close the failing worker test, the bridge bound, and the timeline overflow, and that distance is most of the way closed; the work underneath those items was never the problem.
